79-04 智卷云后端架构设计方案

项目技术栈

  • 后端：JDK21+SpringBoot3.3.4
  • 数据库：MySQL8.0+Redis
  • 消息队列：RabbitMQ

服务发现&服务注册 Nacos

在项目中我们采用本地源码运行Nacos的形式进行部署服务发现和服务注册。

AI访问工具类

架构设计思想

  1. 策略模式：支持不同AI服务的灵活切换
  2. 工厂模式：统一客户端创建入口
  3. 线程池管理：内置可配置的异步执行器
  4. 动态配置：支持运行时API参数变更
  5. 泛型支持：适应不同AI服务的请求/响应结构

好的，我们可以进一步优化你的 AI 访问工具类，使其符合以下设计思想：

  1. 单例模式（Singleton Pattern）
    • 保证 AIClientFactory 和 AIRequestExecutor 只有一个实例，减少资源消耗。
  2. 工厂模式（Factory Pattern）
    • 统一管理 AIClient 的创建，支持动态切换。
  3. 策略模式（Strategy Pattern）
    • 不同 AI 平台的访问逻辑独立封装，方便扩展。
  4. 线程池管理（Thread Pool）
    • 使用 ExecutorService 进行并发控制，提高访问性能。

优化点总结

  1. 单例模式（Holder 静态内部类 + 饿汉单例）：
    • DeepSeekClient 和 ChatGPTClient 采用静态内部类保证线程安全的单例。
    • AIClientFactory 和 AIRequestExecutor 采用饿汉单例，确保唯一实例。
  2. 工厂模式：
    • AIClientFactory 统一管理 AIClient，支持动态注册和扩展。
  3. 策略模式：
    • 不同 AI 平台有独立实现，符合开闭原则（OCP）。
  4. 线程池优化：
    • 采用 FixedThreadPool 提高 AI 访问的并发能力，避免频繁创建线程的开销。

这样，我们的 AI 访问工具类 既高效，又灵活可扩展，能够适应未来的 AI 平台扩展需求 🚀。

关于试卷生成服务把什么作为消息进行传输相关方案提出与确定

背景

背景：用户会在前端页面下单一个试卷生成订单（包含学科-年级等一下基础信息），然后其对应（试卷对外服务）后端会把这个订单保存到数据库中，然后把这个订单ID或者这个订单交给消息队列，由试卷大纲服务负责生成试卷大纲，然后会把这个试卷大纲插入到数据库中，然后把这个试卷大纲ID或者试卷大纲的详细内容交给消息队列，由试卷生成服务对此进行消费，然后获得试卷的详细题目之后，保存到数据库中，最后由试卷对外服务对此进行返回。 现在针对上面的这个背景，我有如下几个方案：

  1. 关于把什么作为消息进行传输，这种方案是把整个内容作为消息传输，那么这个是否存在IO传输消耗比较大的问题；
  2. 把需要传输的消息在数据库中的ID作为消息进行传输，但是这样就会导致访问数据库次数变多，是否影响性能；
  3. 还是把ID作为消息传输，但是会把消息具体的内容保存到Redis中，后续无需访问数据库，直接访问Redis即可；

方案分析与优化建议

方案1：传输完整内容

实现方式：将订单/大纲的完整JSON数据作为消息体传输

优点：
• 数据完整性：消费者无需额外查询即可处理，减少数据库访问次数
• 实时性高：处理链路无额外延迟（适合低延迟场景）
• 架构简单：无需维护额外存储（如Redis）

缺点：
• 网络IO压力大：消息体积大（尤其含题目内容时），占用带宽高
• 序列化开销：JSON序列化/反序列化消耗CPU资源
• 数据冗余：多服务传递相同数据（如试卷基础信息）

适用场景：数据量小（单消息<10KB）、高频字段少、对延迟敏感的场景

优化方向：
• 使用 Protocol Buffers 替代JSON（体积减少30%-50%）
• 压缩消息体（如GZIP）
• 只传输必要字段（去除冗余字段）

方案2：传输数据库ID

实现方式：消息中只含数据库主键ID，消费者自行查询
优点：
• 消息体积小：极大降低网络传输压力
• 数据一致性：总是获取数据库最新状态
• 无缓存维护：架构复杂度低

缺点：
• 数据库压力大：高频访问导致连接池瓶颈（如QPS>5000）
• 延迟增加：每次处理需额外数据库查询（增加10-50ms）
• 级联故障风险：数据库抖动直接影响消息处理

适用场景：数据量大但数据库负载可控、对数据实时性要求极高的场景

优化方向：
• 使用 数据库连接池 和 读写分离
• 批量查询优化（如一次查询多个ID）
• 添加 本地缓存（如Caffeine，缓存短期数据）

方案3：传输ID + Redis缓存

实现方式：消息含ID，数据预存Redis，消费者从Redis读取
优点：
• 性能均衡：Redis吞吐量高（10万+/秒），降低数据库压力
• 可控延迟：Redis查询通常<5ms，性能稳定
• 数据复用：同一数据多次消费无需重复查询

缺点：
• 缓存一致性：需处理数据库与Redis的数据同步（如双写策略）
• 架构复杂度：需维护Redis集群，增加运维成本
• 冷启动问题：新ID首次查询仍需访问数据库

适用场景：高频访问（QPS>1000）、数据更新频率低、允许短暂延迟的场景

优化方向：
• 使用 Redis Hash 结构存储关联数据
• 设置 合理的TTL（如30分钟自动过期）
• 采用 旁路缓存策略（Cache-Aside Pattern）

其他可选方案

方案4：CDC（Change Data Capture） + 事件溯源

实现方式：

  1. 使用 Debezium 监听数据库binlog
  2. 自动捕获数据变更并发布到消息队列
  3. 消费者直接使用变更事件

优点：
• 实时同步：毫秒级数据同步
• 解耦应用：无需业务代码维护数据发布
• 数据溯源：完整记录变更历史

缺点：
• 架构复杂：需维护CDC组件
• 学习成本高：需熟悉流处理技术栈

方案5：混合传输策略

实现方式：
• 小数据（如ID、状态字段）：直接传输完整内容
• 大数据（如题目详情）：传输ID + Redis缓存
• 动态选择：根据数据体积阈值自动切换策略

优点：
• 灵活适配：兼顾性能与实时性
• 资源优化：按需使用不同传输方式

缺点：
• 逻辑复杂：需在业务层实现策略路由
• 维护成本：需监控数据特征变化

方案选型推荐

推荐方案：方案3（ID + Redis）为主，方案1为辅

对比维度 方案3优势
性能 Redis吞吐量远高于数据库，适合高并发场景
可扩展性 通过Redis分片和集群易扩展
数据一致性 通过TTL和主动刷新策略平衡实时性与缓存有效性
成本 虽然增加Redis，但降低数据库扩容需求，总体TCO（总拥有成本）更低

    • 缓存加载耗时（P99 < 20ms）
    • 数据库查询QPS（确保在安全水位下）

总结对比表

完整内容传输 高 低 最高 低 小数据、低延迟需求
方案 网络开销 数据库压力 实时性 架构复杂度 适用场景
纯ID传输 最低 高 低 低 数据库负载低、强一致性要求
ID+Redis 低 中 中 中 高并发、允许短暂延迟
CDC 中 低 高 高 需要数据同步、有运维能力
混合策略 可变 可变 可变 高 业务场景复杂、需灵活适配

根据业务特征选择最适合的方案，推荐优先实施 方案3 并监控核心指标，根据运行情况逐步优化。

最后方案为

  1. 对于试卷对外服务->试卷大纲生成服务：采用直接的消息进行传输（因为试卷配置并不大）;
  2. 对于试卷大纲服务->试卷生成服务：采用ID+Redis进行；
  3. 对于试卷生成服务->试卷对外服务：采用CDC（Change Data Capture） + 事件溯源（学习探讨新技术）
