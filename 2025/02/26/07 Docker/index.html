07 Docker

什么是Docker

当我们谈论 Docker 时，你可以把它想象成一个巨大的、现代化的海运集装箱。这些集装箱可以装载各种各样的货物，从家具到电子设备再到食品，都可以放进去。而 Docker 则是一种类似于这样的集装箱，但是它装载的不是实物货物，而是软件应用程序及其所有必要的组件。

现在，让我们进一步解释一下这个类比。想象一下，你正在开发一个新的软件应用程序，它需要依赖于特定的操作系统、程序库和其他软件组件。在以前，为了确保你的应用程序在不同的计算机环境中能够正常运行，你可能需要手动安装和配置所有这些组件，这就像是把家具逐个放进货舱里，需要大量的时间和精力。

但是，有了 Docker，情况就完全不同了。你可以把你的应用程序和所有它需要的组件都放进一个 Docker 集装箱中。这个集装箱包括了你的应用程序、操作系统、程序库、配置文件等等，就像是把所有的货物都装进了一个标准化的集装箱里面一样。

更棒的是，Docker 集装箱是标准化的，这意味着它们可以在任何支持 Docker 的计算机上运行，就像海运集装箱可以在任何支持海上运输的港口中被卸载和运输一样。这让你的应用程序在不同的环境中更加可移植和可部署。

此外，Docker 还提供了一种称为 Docker 容器的技术，这些容器实际上就是运行着 Docker 集装箱中应用程序的实例。这就好比是在集装箱中放置了货物后，需要有人来处理货物的分拣、装卸等工作一样。

Docker是一个开源的应用容器引擎，让开发者可以打包他们的应用以及依赖包到一个可抑制的容器中，然后发布到任何流行的Linux机器上，也可以实现虚拟化。容器完全使用沙盒机制，相互之间不会存在任何接口。几乎没有性能开销，可以很容易的在机器和数据中心运行。最重要的是，他们不依赖于任何语言、框架或者包装系统。

镜像原理

上面我们把docker比喻成许多的集装箱，那么镜像就是某一个具体的集装箱，每一个镜像里面都装了我们需要的应用程序、运行的环境、依赖项、配置文件以及启动脚本的命令。如果某人需要用到某个东西，直接把这个集装箱给他就好了，就可以直接使用，不需要去配置什么环境之类的东西，很方便。

上面是便于理解的解释，那么什么是镜像呢？镜像是一种轻量级的、可执行的独立软件包。用来打包软件运行环境和基于运行环境的开发软件，它包含运行某个软件所需要的内容，包括代码、运行时、库、环境变量和配置文件。

Docker镜像加载原理

所以和虚拟机相比，为什么docker运行速度更快以及占用的内存较少呢？从上面的图中也可以发现，在docker镜像可以共享底层的基础层。这意味着如果多个容器都使用相同的基础镜像，它们可以共享这些不变的层，减少了磁盘空间的占用，并且在容器启动时也不需要重新加载这些共享的层，从而提高了启动速度。这只是其中的一个原因。

容器

容器是一种轻量级、独立、可移植的软件运行时环境，用于在其中运行应用程序和服务。它们通过将应用程序及其所有依赖项（例如库、环境变量、配置文件等）打包在一起，创建一个隔离的环境，使应用程序能够在不同的计算机环境中以一致的方式运行。

不明白？那换种说法。

  • 镜像就像是三明治的配方：它告诉你需要什么原料，以及如何将它们组合在一起。比如，三明治的配方可能包括面包、火腿、生菜和番茄等原料，以及如何将它们叠放在一起。这就是镜像，它定义了应用程序及其依赖的静态文件和配置信息，但没有实际运行状态。
  • 容器就像是制作好的三明治：一旦你根据配方将原料组合在一起，你就得到了一份真正的三明治。这个三明治是你可以拿在手里吃的，就像容器中运行的应用程序一样。容器包含了镜像中定义的一切，同时还有运行时状态，比如应用程序的内存使用情况、网络连接等。

这样是不是很容易理解了，所以容器是什么？容器是具体的集装箱，镜像是什么？是写好集装箱要装什么的指南。

Dockerfile

Dockerfile是一个创建镜像所有命令的文本文件，包含了一条条指令和说明, 每条指令构建一层,，通过docker build命令，根据Dockerfile的内容构建镜像，因此每一条指令的内容, 就是描述该层如何构建。有了Dockefile,，就可以制定自己的docker镜像规则,只需要在Dockerfile上添加或者修改指令,，就可生成docker 镜像。

在这里我们来整理一下docker容器、dockerfile、docker镜像的关系：

  1. dockerfile是面向开发的，发布项目做镜像的时候就要编写dockerfile文件。
  2. dockerfile：构建文件，定义了一切的步骤，源代码。
  3. dockerImanges：通过dockerfile构建生成的镜像，最终发布和运行的产品。
  4. docker容器：容器就是镜像运行起来提供服务的。

Docker网络

我们使用ifconfig可以看到三组网络。 首先是docker0，这是我们本节的重点，docker的网络。之后是eth0，本机的外网地址。lo口，本地环回地址，可以代表localhost。

为什么Docker需要分配虚拟的IP呢？

  1. 隔离性和独立性：每个容器都被设计为一个独立的运行环境，与宿主主机和其他容器隔离开来。使用虚拟的 IP 地址可以确保每个容器有自己的网络标识，不会与宿主主机或其他容器的网络资源发生冲突。
  2. 便于扩展和迁移：使用虚拟的 IP 地址可以使容器更容易地进行扩展和迁移。当容器需要迁移到不同的主机或者加入到不同的网络中时，只需要调整 IP 地址即可，而不会影响容器内部的配置和程序逻辑。
  3. 方便的服务发现：使用虚拟 IP 地址可以方便地进行服务发现和管理。容器可以通过 DNS 或者其他服务发现机制来自动发现彼此，而不需要硬编码特定的 IP 地址和端口号。
  4. 避免端口冲突：如果每个容器都使用宿主主机的 IP 地址加端口号来进行通信，可能会导致端口冲突和管理上的混乱。而使用虚拟 IP 地址可以避免这种情况，每个容器都可以拥有自己独立的端口空间。

所以最主要的与原因是为了不同的容器之间可以通信以及容器可以和宿主机器保持隔离性，这就是为什么要为docker设置虚拟的ip地址，那样我们之前配置的docker端口号和宿主的端口号一致，却不会冲突这就解释的通了，这是因为宿主的IP和docker容器的IP不是同一个。

Docker网络模式

Docker默认提供了3种网络模式，生成容器时不指定网络模式下默认使用bridge桥接模式。 使用命令查看当前Docker所有的网络模式。

host模式

如果容器启用host模式，那么容器就不会分配一个虚拟的IP，而是和宿主机器使用同一个IP地址。容器将不会虚拟出自己的网卡，配置自己的IP等，而是使用宿主机的IP和端口。但是，容器的其他方面，如文件系统、进程列表等还是和宿主机隔离的。只是IP不隔离了。

使用host模式的容器可以直接使用宿主机的IP地址与外界通信，容器内部的服务端口也可以使用宿主机的端口，host最大的优势就是网络性能比较好，但是docker host上已经使用的端口就不能再用了，网络的隔离性不好。

container模式

这个模式指定新创建的容器和已经存在的一个容器共享一个 Network Namespace，而不是和宿主机共享。新创建的容器不会创建自己的网卡，配置自己的 IP，而是和一个指定的容器共享 IP、端口范围等。同样，两个容器除了网络方面，其他的如文件系统、进程列表等还是隔离的。两个容器的进程可以通过 lo 网卡设备通信。

none模式

使用none模式，Docker容器拥有自己的Network Namespace，但是，并不为Docker容器进行任何网络配置。也就是说，这个Docker容器没有网卡、IP、路由等信息。需要我们自己为Docker容器添加网卡、配置IP等。这种网络模式下容器只有lo回环网络，没有其他网卡。none模式可以在容器创建时通过–network=none来指定。这种类型的网络没有办法联网，封闭的网络能很好的保证容器的安全性。

bridge模式

bridge模式是docker的默认网络模式，不写–net参数，就是bridge模式。使用docker run -p时，docker实际是在iptables做了DNAT规则，实现端口转发功能。可以使用iptables -t nat -vnL查看。bridge模式如下图所示：

当Docker进程启动时，会在主机上创建一个名为docker0的虚拟网桥，此主机上启动的Docker容器都会连接到这个虚拟网桥上。虚拟网桥的工作方式和物理交换机类似，这样主机上的所有容器就通过交换机连在了一个二层网络中。
