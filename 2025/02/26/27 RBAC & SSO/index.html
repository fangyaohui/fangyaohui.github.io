27 RBAC & SSO

参考资料

  1. RBAC——基于角色权限的模型；
  2. 单点登录(SSO)详解；
  3. 深入浅出Cookie、Session、Token：背后的技术原理

RBAC？

Role-Based Access Control，中文意思是：基于角色（Role）的访问控制。这是一种广泛应用于计算机系统和网络安全领域的访问控制模型。

简单的理解就是：通过将权限分配给➡角色，再将角色分配给➡用户，来实现对系统资源的访问控制。

特别注意，在这种模型中，用户与角色之间，角色与权限之间，一般者是多对多的关系。

  1. 角色（Role）：角色是指在系统中具有一组相关权限的抽象概念，代表了用户在特定上下文中的身份或职能，例如管理员、普通用户等。
  2. 权限（Permission）：权限是指对系统资源进行操作的许可，如读取、写入、修改等。权限可以被分配给角色。
  3. 用户（User）：用户是指系统的实际使用者，每个用户可以被分配一个或多个角色。
  4. 分配（Assignment）：分配是指将角色与用户关联起来，以赋予用户相应的权限。

Why？

很简单：方便开发。

因为当用户的数量非常大时，要给系统每个用户逐一授权，是件非常烦琐的事情。这时，就需要给用户分组，每个用户组内有多个用户。除了可给用户授权外，还可以给用户组授权。这样一来，用户拥有的所有权限，就是用户个人拥有的权限与该用户所在用户组拥有的权限之和。

SSO？

单点登录（Single Sign On），简称为 SSO，是目前比较流行的企业业务整合的解决方案之一。SSO的定义是在多个应用系统中，用户只需要登录一次就可以访问所有相互信任的应用系统（例如淘宝、天猫之间的关系）。简而言之，多个系统，统一登陆。

单点登录实现方式

一般有三种实现方式：

  1. Cookie
  2. Session
  3. Token

Cookie

在用户登录成功之后，后端生成一个随机字符串，将这个随机字符串存放在Cookie中返回给前端页面，前端页面每次访问时，需要携带这个Cookie给后端。同时后端将这个随机字符串作为Key，用户ID作为Value存放在Redis中，然后后端每次根据前端传来的Cookie来获取这个随机字符串，然后从Redis中获取当前用户ID。

缺点：

（1）Cookie不安全：攻击者可以通过伪造Cookie伪造成特定用户身份，可以用加密规避（前提是加密算法不被泄露）；由于 Cookie 数据存储在客户端，因此存在被用户或第三方篡改的风险。虽然可以通过加密和签名等手段提高安全性，但仍然存在一定的风险；

（2）不能跨域实现免登：Cookie一般是以域名为分割的。如：baidu.com 无法读到 cnblogs.com 写的 cookie。域名是可以访问上级域名的Cookie的。 即a.baidu.com是可以访问baidu.com下的Cookie的。（一级域名相同，只是二级或更高级域名不同的站点，如：a.baidu.com、b.baidu.com可以通过设置 domain 参数共享 cookie 读写。）

Session

Session是保存在服务端的，所以如果是分布式系统的话，那么就会出现服务器不共享Session的问题。如果是分布式系统，session属于有状态，实现分布式session有四种方案（session复制、客户端存储、HASH一致性、统一存储）。详情请看：分布式session解决方案_半格咖啡的博客-CSDN博客。

缺点：

（1）服务器压力增大：通常session是存储在内存中的，每个用户通过认证之后都会将session数据保存在服务器的内存中，而当用户量增大时，服务器的压力增大。（可以将数据保存在磁盘中）

（2）扩展性不强：如果将来搭建了多个服务器，虽然每个服务器都执行的是同样的业务逻辑，但是session数据是保存在内存中的（不是共享的），用户第一次访问的是服务器1，当用户再次请求时可能访问的是另外一台服务器2，服务器2获取不到session信息，就判定用户没有登陆过。（可以使用分布式session将session在各个集群中保持一致）

（3）CSRF跨站伪造请求攻击：session是基于cookie进行用户识别的, cookie如果被截获，用户就会很容易受到跨站请求伪造的攻击。

CSRF（Cross-Site Request Forgery，跨站请求伪造）攻击是一种针对Web应用程序的攻击方式，它利用用户在已登录的Web应用程序上的有效会话（Session）来发送恶意请求。

Token

Token 是令牌的意思，由服务端生成并发放给客户端，是一种具有时效性的验证身份的手段。

Token 避免了 Session 机制带来的海量信息存储问题，也避免了 Cookie 机制的一些安全性问题，在现代移动互联网场景、跨域访问等场景有广泛的用途。

机场项目后台管理系统登录流程

通过设置一个中央登录注册中心，该中心负责生成Token并保存到cookie中返回给浏览器。该浏览器得到这个cookie后跳转到子应用，同步检测本地、本应用服务端是否有该Token保持的登录信息，如果都没有则会访问登录注册服务中心，该服务中心会解析这个Token并同时判断该Token是否有效，有效后返回用户的登录具体信息，前端页面将会保存一份到本地，之后每次请求也会将该信息保存到本地后端服务器中。
