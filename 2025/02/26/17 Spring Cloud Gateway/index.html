什么是服务网关

在服务网关出现之前，我们的业务还是基于传统的单体架构，即我们的业务都放在一个微服务中。随着计算机的发展，我们的业务也越来越多，这时候如果把所有业务都放在一个微服务中，很难去维护，那么就提出了微服务架构，即把一个系统划分成多个微服务。

但是把系统划分多个微服务后会出现一个比较麻烦的问题。前端调用一个后端接口时就需要知道这个接口在哪个微服务，这个微服务的地址是什么？我们可以这样理解，客户端的请求就是一个人带着消息来到了我们的小区，但是这个消息该给谁呢？这个人得去自己寻找地址，这个工作量很大。

这时候人们就提出了网关的概念，那么什么是网关呢？我们简单的理解，网关就是我们小区门口的门卫。当客户端的请求，即这个请求人带着消息来到了我们小区，询问门卫这个消息该给谁，门卫一看消息头，就直接把地址告知了这个请求人。这样是不是简单了些。

那么这样前端的所有请求都可以直接给这个网关（门卫），简化了客户端的工作。

前言

分布式开发时，微服务会有很多，但是网关是请求的第一入口，所以一般会把客户端请求的权限验证统一放在网关进行认证与鉴权。SpringCloud Gateway 作为 Spring Cloud 生态系统中的网关，目标是替代 Zuul，为了提升网关的性能，SpringCloud Gateway是基于WebFlux框架实现的，而WebFlux框架底层则使用了高性能的Reactor模式通信框架Netty。

在SpringGateway中已经依赖了Web Flux，所以才会有gateway中不能依赖Web，否则会有两个Web从而造成冲突。

如果需要在gateway中引入Security，那么Security就需要使用Web Flux来实现。

首先需要配置@EnableWebFluxSecurity注解，开启Spring WebFlux Security的支持

流程

执行的先后顺序：

  1. ScSecurityContextRepository 从请求头中读取Token放入AuthenticationToken中；
  2. ScAuthenticationManager 读取Token验证当前Token是否有效正确；
  3. ScAuthorizationManager 验证当前用户是否有权限访问该资源；
  4. 如果第2和第3没有权限访问，则会执行对应的失败处理逻辑；
  5. ScFilter Security 过滤器；
  6. gateway自定义的过滤器；
  7. 转发请求。

总结

在SpringGateway引入SpringSecurity的整个流程中，我们需要定义Security服务，也就是定义一个Security配置类，接着定义从请求头中读取Token并放入AuthenticationToken中的服务，然后定义认证和授权处理逻辑，接着定义认证和授权失败处理逻辑，最后定义拦截器，整个流程就很清晰了。

在请求网关获取图片的时候，有时可以成功，有时失败，看图中最下面同样的一个请求却又请求成功了。

到网关去查看原因，网关报错如下：

开始以为是前端设置的过期时间太短了，就重新设置，无果，故开始查询相关资料来找到该原因。

搜索

一篇博客：reactor.netty.http.client.PrematureCloseException: Connection prematurely closed BEFORE response解决方案

问题不在于SpringCloudGateway，而是在于Reactor-Netty：

这里使用一个Spring Boot内置Tomcat作为服务提供方，用户通过SCG访问，SCG代理请求。

默认情况下，SCG内部创建的连接是不会被回收的，一直存在于内存中，而Spring Boot内置的Tomcat不一样，默认在20s之后没有数据交互，便会回收掉这个连接，在回收的时候恰巧碰到又来了请求，刚好又在SCG拿到这个连接来尝试请求Tomcat，就会出现这个异常。

所以，不要指望在Reactor-Netty或是SCG中解决这个问题，这需要网关和后端服务配合解决，最大限度不出现这个异常。
