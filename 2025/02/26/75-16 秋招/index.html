1. WebSocket 和 HTTP 的区别是什么？在什么情况下应该使用 WebSocket 而不是 HTTP？

（1）基础概念

  • HTTP 是 短连接（请求-响应模型），每次请求都需要客户端主动发起，服务器处理完请求后返回数据并关闭连接。
  • WebSocket 是 长连接（双向通信），建立连接后，客户端和服务器可以互相主动发送消息，不需要每次都重新建立连接。

（2）核心区别

特性 HTTP WebSocket
连接方式 短连接，每次请求重新建立连接 长连接，建立后保持连接
通信方向 单向，客户端主动请求，服务器响应 双向，服务器和客户端都可以主动发送消息
协议 基于 请求-响应 机制 基于 双工通信 机制
报文格式 HTTP 头部 + 负载（文本/JSON/二进制） 轻量二进制帧（帧格式优化）
效率 每次请求需要重新建立 TCP 连接（高开销） 连接建立后可多次交互（低开销）
适用场景 普通的网页请求、API 调用 即时聊天、游戏、股票行情、推送通知

（3）什么时候使用 WebSocket？

  • 适用于高频、低延迟的实时应用
    • 即时通讯（如聊天软件、客服系统）
    • 多人在线游戏（如棋牌、MMO）
    • 实时推送（如微博热搜、天气、新闻推送）
    • 股票交易（如行情数据推送）
    • 协作应用（如 Google Docs 共享文档编辑）

（4）为什么不总是用 WebSocket？

  • WebSocket 连接需要维护会话，可能占用更多服务器资源
  • 并不是所有场景都需要实时双向通信
  • 某些浏览器或防火墙可能不支持 WebSocket

2. WebSocket 是基于 TCP 还是 UDP 的？它是如何保证消息的有序性和可靠性的？

（1）WebSocket 是基于 TCP 的

  • WebSocket 本质上是 基于 TCP 之上的一种 双向通信协议。
  • 它不是 UDP，所以 WebSocket 不能天然保证低延迟和丢包容忍性，但可以通过 优化数据帧格式 降低 TCP 的开销。

（2）如何保证消息的有序性？

  • TCP 本身是有序的（通过 序列号 + 滑动窗口 + ACK 确认机制）。
  • WebSocket 直接 继承 TCP 的有序性，所以 WebSocket 保证了消息的顺序，不会乱序。

（3）如何保证消息的可靠性？

  • 基于 TCP 的三次握手、超时重传、ACK 机制 保证了数据不会丢失。
  • WebSocket 层面 还可以增加应用级别的 消息确认机制（ACK 确认），如果客户端未收到消息，可以请求服务器重新发送。

3. WebSocket 的握手过程是怎样的？如何从 HTTP 连接升级到 WebSocket 连接？

（1）WebSocket 连接是如何建立的？

WebSocket 需要先通过 HTTP 请求 进行 协议升级，然后变成 WebSocket 连接：

  1. 客户端发送 HTTP 请求（带 Upgrade 头部）

    • Upgrade: websocket → 告诉服务器要升级 WebSocket 连接。
    • Connection: Upgrade → 让 HTTP 连接变成 WebSocket。
    • Sec-WebSocket-Key → 用于校验，防止中间人攻击。

  2. 服务器返回 WebSocket 握手响应

    • 101 Switching Protocols → 表示协议升级成功。
    • Sec-WebSocket-Accept → 服务器计算 Sec-WebSocket-Key 并返回，保证安全性。

  3. 握手完成后，HTTP 连接升级为 WebSocket 连接

    • 之后 客户端和服务器可以自由发送 WebSocket 帧，不需要 HTTP 头部。

（2）WebSocket 协议为什么要先经过 HTTP 进行握手？

  • 兼容性：在 WebSocket 诞生前，浏览器主要支持 HTTP，HTTP 握手可以兼容 现有的代理和服务器。
  • 安全性：Sec-WebSocket-Key 防止 中间人攻击，确保连接是合法的 WebSocket 连接。
  • 方便负载均衡：很多 Web 服务器（如 Nginx）可以通过 HTTP 代理 WebSocket。

4. WebSocket 连接的 ping/pong 机制是什么？为什么需要它？

（1）什么是 ping/pong 机制？

WebSocket 定义了一个 心跳机制（ping / pong），用于 检测 WebSocket 连接是否存活。

  • 服务器可以发送 ping，客户端必须在 规定时间内回复 pong。
  • 如果客户端不回复 pong，服务器就会关闭连接，防止死连接占用资源。

（2）为什么需要 ping/pong？

  1. 检测 WebSocket 连接是否仍然存活
    • WebSocket 是 长连接，如果客户端意外断线（如网络中断、浏览器崩溃），服务器 无法主动感知，需要 ping/pong 确认连接状态。
  2. 防止 TCP 连接被中间设备（防火墙、NAT）关闭
    • 如果 WebSocket 长时间没有数据传输，防火墙可能会 自动断开 TCP 连接，ping/pong 机制可以 保持连接存活。
  3. 减少无效的 WebSocket 连接
    • 服务器可以定期 ping，如果发现客户端没有响应 pong，就主动关闭 WebSocket，减少服务器资源占用。

（1）WebSocket vs STOMP

特性 WebSocket STOMP (Simple Text Oriented Messaging Protocol)

1. WebSocket 连接较多的情况下（如 10w 以上），如何优化 WebSocket 服务器的性能？

1.1 WebSocket 服务器的性能瓶颈

在高并发 WebSocket 服务器中，主要的性能瓶颈包括：

  • 连接管理开销：每个 WebSocket 连接都需要维护 Session，占用一定的 内存 和 CPU 资源。
  • I/O 处理能力：WebSocket 需要 高效的 I/O 处理模型，否则会导致吞吐量下降。
  • 消息广播与分发：在 群聊或消息推送 场景下，消息的分发会导致 CPU、网络带宽 开销过高。
  • 负载均衡：单台服务器的连接数上限受限，如何水平扩展非常关键。

1.2 性能优化方案

1.2.1 选择合适的 WebSocket 服务器
  • Netty：相比于 Tomcat、Spring 默认的 WebSocket 实现，Netty 更适合 高并发 WebSocket 连接，因为它采用 异步 I/O（NIO），减少线程阻塞，提高吞吐量。
  • Vert.x WebSocket：基于 Netty，适用于高并发 WebSocket 场景。
1.2.2 限制每个连接的资源占用
  • 减少 Session 内存占用：避免直接在 JVM 内存中存储过多 WebSocket 连接，可以使用 Redis / Zookeeper 进行管理。
  • 消息队列优化：避免服务器缓存大量未发送的消息，使用 Kafka 或 Redis 进行消息缓冲，异步推送消息。
1.2.3 使用异步非阻塞 I/O（Reactor 模型）
  • Tomcat 默认 WebSocket 采用 BIO（阻塞 I/O），对于高并发不友好。
  • Netty + Reactor（如 Spring WebFlux） 采用 事件驱动的异步处理，可以大幅提高吞吐量。
1.2.4 负载均衡（集群部署）
  • Nginx + WebSocket 反向代理：使用 sticky session（会话保持），确保同一 WebSocket 连接始终路由到同一台服务器。
  • Kubernetes + Ingress：适用于云原生架构，支持 WebSocket 的负载均衡。
1.2.5 消息压缩和优化
  • WebSocket 压缩扩展（PMCE）：减少 WebSocket 消息体积，提高传输效率。
  • JSON → Protobuf 或 MessagePack：减少数据大小，提高解析速度。

2. WebSocket 适合用来做消息持久化存储吗？如果需要实现聊天记录持久化，该如何设计？

2.1 WebSocket 不适合直接做持久化存储的原因

  • WebSocket 是实时通信协议，适合 低延迟传输，但 不适合数据持久化。
  • 服务器重启或故障时，WebSocket 连接会丢失，导致数据不可追溯。
  • 数据库写入会影响吞吐量，如果每条消息都直接写入数据库，会严重影响性能。

前后端页面所有请求和响应都进行加密处理

在前后端加密处理过程中，首先需要选择合适的加密算法。根据加密方式的不同，加密算法分为两大类：对称加密和非对称加密。其分类标准为公钥和私钥是否相同，若相同则为对称加密，反之则为非对称加密。

常见的对称加密算法有 AES，而非对称加密算法通常采用 RSA。详细的加密算法介绍可参考[62 加密算法](https://fangyaohui.github.io/2025/02/26/62 加密算法/)。

在实际应用中，常见的做法是使用 RSA 加密算法进行密钥交换，成功交换后，使用 AES 算法进行数据加密和解密。

本项目中，考虑到性能与易用性，选择了 AES 作为加密算法。

为支持这一加密方式，开发了一个模块，封装了对接口请求进行解密与响应加密的功能，方便未来扩展和维护。

在前后端加密的实现中，常见的两种情形是：一是对请求 URL 中的路径进行加密；二是对请求体进行加密而不对 URL 进行加密。若选择第一种方案，则后端服务需要在网关层解密请求路径并处理请求体，之后将请求转发至目标接口；若选择第二种方案，则无需在网关层处理，直接将请求转发至目标接口。

接下来，详细分析在加密解密过程中可能遇到的技术难点。

首先，假设一个请求的 URL 被加密，经过网关解密后，若请求体未解密，且在目标接口处理时，Spring Boot 会自动进行参数匹配，此时会导致类型不匹配的错误，无法进行自动注入。

其次，HttpServletRequest 默认情况下的输入流只能读取一次，因此如果需要多次访问请求体的内容，需要使用 ContentCachingRequestWrapper 进行请求体缓存。此设计特别适用于需要多次读取请求体内容的场景，例如在 AOP 中解密请求体内容。通过自定义过滤器，将原始的 HttpServletRequest 封装为 ContentCachingRequestWrapper，从而支持多次读取请求体内容。

上述是加密解密过程中的第一个技术难点。

第二个难点是处理 PO 实体类中属性的加解密。部分属性可能需要指定进行加解密，这将导致层层嵌套。虽然基础数据类型的加解密相对简单，但对于引用类型的数据，则需要递归查找被注解标记的属性，并进行相应的加解密操作。如果该属性是引用类型，则其引用的属性也需进行解密。因此，基于策略工厂等设计模式，可以灵活实现这一需求。以下是加解密策略设计的代码示例：
