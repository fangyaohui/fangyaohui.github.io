1. WebSocket 和 HTTP 的区别是什么？在什么情况下应该使用 WebSocket 而不是 HTTP？

（1）基础概念

  • HTTP 是 短连接（请求-响应模型），每次请求都需要客户端主动发起，服务器处理完请求后返回数据并关闭连接。
  • WebSocket 是 长连接（双向通信），建立连接后，客户端和服务器可以互相主动发送消息，不需要每次都重新建立连接。

（2）核心区别

特性 HTTP WebSocket
连接方式 短连接，每次请求重新建立连接 长连接，建立后保持连接
通信方向 单向，客户端主动请求，服务器响应 双向，服务器和客户端都可以主动发送消息
协议 基于 请求-响应 机制 基于 双工通信 机制
报文格式 HTTP 头部 + 负载（文本/JSON/二进制） 轻量二进制帧（帧格式优化）
效率 每次请求需要重新建立 TCP 连接（高开销） 连接建立后可多次交互（低开销）
适用场景 普通的网页请求、API 调用 即时聊天、游戏、股票行情、推送通知

（3）什么时候使用 WebSocket？

  • 适用于高频、低延迟的实时应用
    • 即时通讯（如聊天软件、客服系统）
    • 多人在线游戏（如棋牌、MMO）
    • 实时推送（如微博热搜、天气、新闻推送）
    • 股票交易（如行情数据推送）
    • 协作应用（如 Google Docs 共享文档编辑）

（4）为什么不总是用 WebSocket？

  • WebSocket 连接需要维护会话，可能占用更多服务器资源
  • 并不是所有场景都需要实时双向通信
  • 某些浏览器或防火墙可能不支持 WebSocket

2. WebSocket 是基于 TCP 还是 UDP 的？它是如何保证消息的有序性和可靠性的？

（1）WebSocket 是基于 TCP 的

  • WebSocket 本质上是 基于 TCP 之上的一种 双向通信协议。
  • 它不是 UDP，所以 WebSocket 不能天然保证低延迟和丢包容忍性，但可以通过 优化数据帧格式 降低 TCP 的开销。

（2）如何保证消息的有序性？

  • TCP 本身是有序的（通过 序列号 + 滑动窗口 + ACK 确认机制）。
  • WebSocket 直接 继承 TCP 的有序性，所以 WebSocket 保证了消息的顺序，不会乱序。

（3）如何保证消息的可靠性？

  • 基于 TCP 的三次握手、超时重传、ACK 机制 保证了数据不会丢失。
  • WebSocket 层面 还可以增加应用级别的 消息确认机制（ACK 确认），如果客户端未收到消息，可以请求服务器重新发送。

3. WebSocket 的握手过程是怎样的？如何从 HTTP 连接升级到 WebSocket 连接？

（1）WebSocket 连接是如何建立的？

WebSocket 需要先通过 HTTP 请求 进行 协议升级，然后变成 WebSocket 连接：

  1. 客户端发送 HTTP 请求（带 Upgrade 头部）

    • Upgrade: websocket → 告诉服务器要升级 WebSocket 连接。
    • Connection: Upgrade → 让 HTTP 连接变成 WebSocket。
    • Sec-WebSocket-Key → 用于校验，防止中间人攻击。

  2. 服务器返回 WebSocket 握手响应

    • 101 Switching Protocols → 表示协议升级成功。
    • Sec-WebSocket-Accept → 服务器计算 Sec-WebSocket-Key 并返回，保证安全性。

  3. 握手完成后，HTTP 连接升级为 WebSocket 连接

    • 之后 客户端和服务器可以自由发送 WebSocket 帧，不需要 HTTP 头部。

（2）WebSocket 协议为什么要先经过 HTTP 进行握手？

  • 兼容性：在 WebSocket 诞生前，浏览器主要支持 HTTP，HTTP 握手可以兼容 现有的代理和服务器。
  • 安全性：Sec-WebSocket-Key 防止 中间人攻击，确保连接是合法的 WebSocket 连接。
  • 方便负载均衡：很多 Web 服务器（如 Nginx）可以通过 HTTP 代理 WebSocket。

4. WebSocket 连接的 ping/pong 机制是什么？为什么需要它？

（1）什么是 ping/pong 机制？

WebSocket 定义了一个 心跳机制（ping / pong），用于 检测 WebSocket 连接是否存活。

  • 服务器可以发送 ping，客户端必须在 规定时间内回复 pong。
  • 如果客户端不回复 pong，服务器就会关闭连接，防止死连接占用资源。

（2）为什么需要 ping/pong？

  1. 检测 WebSocket 连接是否仍然存活
    • WebSocket 是 长连接，如果客户端意外断线（如网络中断、浏览器崩溃），服务器 无法主动感知，需要 ping/pong 确认连接状态。
  2. 防止 TCP 连接被中间设备（防火墙、NAT）关闭
    • 如果 WebSocket 长时间没有数据传输，防火墙可能会 自动断开 TCP 连接，ping/pong 机制可以 保持连接存活。
  3. 减少无效的 WebSocket 连接
    • 服务器可以定期 ping，如果发现客户端没有响应 pong，就主动关闭 WebSocket，减少服务器资源占用。

2. Spring WebSocket 和 STOMP 协议有什么区别？为什么很多时候 STOMP 适用于 WebSocket？

（1）WebSocket vs STOMP

特性 WebSocket STOMP (Simple Text Oriented Messaging Protocol)
数据格式 只能发送文本/二进制数据 基于 消息帧（有 命令 和 头部）
订阅模型 没有内置订阅机制 内置 订阅/广播 机制（类似消息队列）
认证方式 需要自己实现 STOMP 提供 身份认证机制
适用场景 点对点聊天 群聊、推送通知、实时协作

（2）为什么 STOMP 适用于 WebSocket？

  • WebSocket 是一个协议，但缺少消息格式 → STOMP 解决了这个问题，提供了结构化消息。
  • WebSocket 没有内置订阅功能 → STOMP 提供了 发布/订阅 机制。
  • WebSocket 认证需要自己实现 → STOMP 可以 携带 token 认证。
