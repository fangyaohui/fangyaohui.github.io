体系结构

前言

在我们日常生活中会用浏览器访问各种页面，我们需要在浏览器的窗口输入对应的URL，如下：

如果我们回车就会进入到对应的URL的页面窗口。

那么在这个的过程中，浏览器以及计算机网络做了什么事情呢？现在我们一步一步的进行分析来介绍。

应用层

HTTP

当我们在这个输入框中输入了网址，然后回车之后，浏览器首先要对这个地址进行解析，然后发送给Web服务器进行请求信息。让我们首先来分析一下这个URL的构成：

浏览器会开始解析这个URL确定Web服务器和文件名，并把请求信息进行封装，封装好之后会进入下一阶段，它需要知道这个Web服务器在哪？以及谁能帮他传送。

DNS

我们上面知道了这个请求的Web服务器的URL，我们知道了网址（域名），但是在互联网的世界里不能根据域名来找到这个服务器，就好像无法根据名称来找到这个人所在的地址。

所以我们需要一个映射，这个映射需要记录下当前名称（域名）所对应的地址（IP地址），拥有这个映射表的服务器我们称为DNS服务器。

在这个DNS服务器中，它会根据域名来解析获得该域名的IP地址。

现在我们输入了www.baidu.com，这里有三层级。最高级是谁呢？是com，要特别注意在这里，在外国人的习惯中，从左到右的是从小到大的关系。

根域是在最顶层，它的下一层就是 com 顶级域，再下面是baidu.com。

所以域名的层级关系类似一个树状结构：

  • 根 DNS 服务器
  • 顶级域 DNS 服务器（com）
  • 权威 DNS 服务器baidu.com。

在DNS获得指定域名的IP地址时，会有两种方式。

  1. 递归查询：当收到查询请求报文的域名服务器不知被查询域名的IP地址时，该域名服务器就以DNS客户的身份向下一步应查询的域名服务器发出查询请求，即替本地域名字服务器继续查询，较少使用。

  2. 迭代查询：当收到查询请求报文的域名服务器不知道被查询域名的IP地址时，就把自己知道的下一步应查询的域名服务器IP地址告诉本地域名字服务器，由本地域名字服务器继续向该域名服务器查询，直到得到所要解析的域名的IP地址，或者查询不到所要解析的域名的IP地址，通常使用。

通过上面的过程，我们终于获得了这个域名的IP地址，但是这个IP地址该告诉谁呢？什么时候用到呢？如果用到了该把这个IP地址放在什么地方呢？

应用层常见协议

  1. DNS（域名系统）：DNS 是互联网使用的命名系统，用来把人们使用的机器名字（域名）转换为 IP 地址。域名服务器分为四种类型：根域名服务器、顶级域名服务器、本地域名服务器、权限域名服务器。

  2. FTP（文件传输协议）：FTP 是使用得最广泛的文件传送协议。FTP 提供交互式的访问，常用于局域网的文件存储服务器。

  3. Telnet（远程终端协议）：telnet 是一个简单的远程终端协议，结合端口使用，常用于服务探测。

  4. HTTP（超文本传送协议）：HTTP 是面向事务的应用层协议，它是互联网能够可靠地交换文件的重要基础。http 使用面向连接的 TCP 作为运输层协议，保证了数据的可靠性。

  5. SMTP（电子邮件协议）：SMTP 是简单邮件传送协议。SMTP规定了在两个相互通信的 SMTP 进程之间应如何交换信息。SMTP通信包括三个阶段，分别是：建立连接、邮件传送、连接释放。

  6. POP3（邮件读取协议）：POP3，全称是 Post Office Protocol 3，该协议通常被用来接收电子邮件。

  7. SNMP（简单网络管理协议）：SNMP 协议由三部分组成，包括：SNMP本身、管理信息结构 SMI 和管理信息 MIB。SNMP 定义了管理站和代理之间所交换的分组格式。SMI 定义了命名对象类型的通用规则，以及把对象和对象的值进行编码。MIB 在被管理的实体中创建了命名对象，并规定类型。

  8. TFTP（简单文件传送协议）：TFTP是TCP/IP协议族中的一个用来在客户机与服务器之间进行简单文件传输的协议，它基于UDP协议而实现的，提供不复杂、开销不大的文件传输服务。默认端口号是 69。

  9. 其他应用层协议：除了上面介绍的协议外，其他应用层协议还有 DHCP、Gopher 、IMAP4、IRC、 NNTP、 XMPP、 SIP、 SSH、 RPC、 RTCP、 RTP、RTSP、SDP、 SOAP、 GTP、 STUN、 NTP、SSDP、 BGP 等。

协议栈

上面HTTP获得了IP地址之后，浏览器的传输工作就会交给本机操作系统的协议栈中，由操作系统的协议栈来负责下面的数据传输工作，其主要负责的层有传输层&网络层，下面来具体的介绍一下协议栈中的作用。

运输层

应用层把需要传输的数据封装好后交给运输层进行处理，应用层有很多种协议，每种协议对应使用运输层不同的协议，运输层的主要协议有TCP和UDP。下面列出一些应用层常用协议对应的传输层的协议。

而这里HTTP协议使用的是TCP协议。运输层确定好了使用的协议之后，就会给这个添加请求头在数据包的前面。

TCP

TCP报文头结构：

首先，源端口号和目标端口号是不可少的，如果没有这两个端口号，数据就不知道应该发给哪个应用。

端口号的作用是为了接受到请求时，明确这个报文应该交给应用层的哪一个应用进行处理。

接下来有包的序号，这个是为了解决包乱序的问题。

还有应该有的是确认号，目的是确认发出去对方是否有收到。如果没有收到就应该重新发送，直到送达，这个是为了解决不丢包的问题。

接下来还有一些状态位。例如 SYN 是发起一个连接，ACK 是回复，RST 是重新连接，FIN是结束连接等。TCP 是面向连接的，因而双方要维护连接的状态，这些带状态位的包的发送，会引起双方的状态变更。

还有一个重要的就是窗口大小。TCP 要做流量控制，通信双方各声明一个窗口（缓存大小），标识自己当前能够的处理能力，别发送的太快，撑死我，也别发的太慢，饿死我。

除了做流量控制以外，TCP还会做拥塞控制，对于真正的通路堵车不堵车，它无能为力，唯一能做的就是控制自己，也即控制发送的速度。不能改变世界，就改变自己嘛。

为了保证TCP的可靠性，所以TCP两个连接时会进行三次握手来保证连接的可靠。

为什么要进行三次握手呢？上面说到了是为了保证TCP的可靠性，即是为了保证服务端和客户端都有接受和发送报文的能力。

一开始，客户端和服务端都处于 CLOSED 状态。先是服务端主动监听某个端口，处于 LISTEN 状态。

然后客户端主动发起连接 SYN，之后处于 SYN-SENT 状态。

服务端收到发起的连接，返回 SYN，并且 ACK 客户端的 SYN，之后处于 SYN-RCVD 状态。

客户端收到服务端发送的 SYN 和 ACK 之后，发送 ACK 的 ACK，之后处于 ESTABLISHED状态，因为它一发一收成功了。

服务端收到 ACK 的 ACK 之后，处于 ESTABLISHED 状态，因为它也一发一收了。

所以三次握手目的是保证双方都有发送和接收的能力。

UDP

UDP的设计理念就是为了尽可能快的发送报文，所以在一定程度上会忽略其可靠性。

UDP的头部包含一下字段：

  • 源端口号（Source Port）：发送方的端口号。
  • 目的端口号（Destination Port）：接收方的端口号。
  • 长度（Length）：UDP报文的长度，包括头部和数据部分。
  • 校验和（Checksum）：用于检测报文在传输过程中是否出现错误。

可以看到并又没保证可靠的字段，唯一的校验和字段也只是为了简单检查报文在传输过程中是否出现了错误。

  1. HTTP (HyperText Transfer Protocol) - 端口 80，用于网页传输。

网络层

IP

IP报文头部信息

  1. 版本（Version） - 4位字段，指定IP协议的版本。IPv4是4，IPv6是6。
  2. 首部长度（Header Length） - 4位字段，指示IP报文头部的长度，以32位字（4字节）为单位。这允许IP头部可以扩展以包含选项。
  3. 服务类型（Type of Service, ToS） - 8位字段，在IPv4中用于指定数据包的服务质量要求，如优先级、延迟、吞吐量等。在IPv6中，这部分被分为两个字段：流量类（Traffic Class）和流标签（Flow Label）。
  4. 总长度（Total Length） - 16位字段，表示整个IP数据包的长度，包括头部和数据部分，单位是字节。
  5. 标识（Identification） - 16位字段，用于唯一标识一个IP数据包。当数据包太大而需要分片时，这个字段帮助接收端重新组装数据包。
  6. 标志（Flags） - 3位字段，用于控制和指示分片的行为。其中，最常用的是MF（More Fragments）标志，表示是否还有更多的分片。
  7. 片偏移（Fragment Offset） - 13位字段，指分片相对于原始数据包开始处的位置，以8字节为单位。
  8. 生存时间（Time to Live, TTL） - 8位字段，指定数据包在网络中可以经过的最大路由器数。每经过一个路由器，TTL减1，当TTL为0时，数据包将被丢弃，以防止无限循环。
  9. 协议（Protocol） - 8位字段，指定了数据包中封装的上层协议类型，如TCP是6，UDP是17。
  10. 头部校验和（Header Checksum） - 16位字段，用于检测头部信息在传输过程中是否出现错误。
  11. 源IP地址（Source Address） - 32位字段（IPv4），表示数据包发送者的IP地址。
  12. 目的IP地址（Destination Address） - 32位字段（IPv4），表示数据包接收者的IP地址。
  13. 选项（Options） - 可选字段，用于包含额外的IP功能，如安全和时间戳等。这个字段的长度由首部长度字段决定。
  14. 填充（Padding） - 确保IP头部是32位字的倍数，如果需要的话，用以填充。

数据链路层

MAC

MAC 头部是以太网使用的头部，它包含了接收方和发送方的 MAC 地址等信息。

在 MAC 包头里需要发送方 MAC 地址和接收方目标 MAC 地址，用于两点之间的传输。

一般在 TCP/IP 通信里，MAC 包头的协议类型只使用：

  • 0800 ： IP 协议
  • 0806 ： ARP 协议（Address Resolution Protocol）

在网络层中我们获得了目标服务的IP地址，但是在数据链路层我们需要根据目标的MAC地址来传送数据包，可是我们如何知道目标地址的MAC地址呢？这就需要ARP协议（地址解析协议）

发送方的 MAC 地址获取就比较简单了，MAC 地址是在网卡生产时写入到 ROM 里的，只要将这个值读取出来写入到 MAC 头部就可以了。

接收方的 MAC 地址就有点复杂了，只要告诉以太网对方的 MAC 的地址，以太网就会帮我们把包发送过去，那么很显然这里应该填写对方的 MAC 地址。

ARP协议通过同一子网进行广播获得目标IP的MAC的地址，但是如果不在同一子网的IP就会被发送到本地子网的默认网关，即路由器的接口地址。这个地址通常通过配置在主机上的默认网关来指定。路由器根据其路由表来决定如何将数据包转发到下一个网络或最终目的地。这个过程在网络层进行。最终，数据包到达包含目标IP地址的子网。一旦数据包到达目标子网，就可以使用ARP来解析目标IP地址对应的MAC地址。这是因为现在目标IP和发起ARP请求的设备位于同一子网内。获得MAC地址后，数据链路层可以构建以太网帧，将数据包作为有效载荷，并将源和目的MAC地址设置为本地网络上的相应设备地址。

所以ARP的使用场景是在同一子网，如果不是在同一子网则不可以使用ARP协议。

物理层

IP 生成的网络包只是存放在内存中的一串二进制数字信息，没有办法直接发送给对方。因此，我们需要将数字信息转换为电信号，才能在网线上传输，也就是说，这才是真正的数据发送过程。

负责执行这一操作的是网卡，要控制网卡还需要靠网卡驱动程序。

网卡驱动从 IP 模块获取到包之后，会将其复制到网卡内的缓存区中，接着会其开头加上报头和起始帧分界符，在末尾加上用于检测错误的帧校验序列。

最后网卡会将包转为电信号，通过网线发送出去。

交换机

下面来看一下包是如何通过交换机的。交换机的设计是将网络包原样转发到目的地。交换机工作在 MAC 层，也称为二层网络设备。

位于同一个交换机下的处于同一子网。

首先，电信号到达网线接口，交换机里的模块进行接收，接下来交换机里的模块将电信号转换为数字信号。

然后通过包末尾的 FCS 校验错误，如果没问题则放到缓冲区。这部分操作基本和计算机的网卡相同，但交换机的工作方式和网卡不同。

计算机的网卡本身具有 MAC 地址，并通过核对收到的包的接收方 MAC 地址判断是不是发给自己的，如果不是发给自己的则丢弃；相对地，交换机的端口不核对接收方 MAC 地址，而是直接接收所有的包并存放到缓冲区中。因此，和网卡不同，交换机的端口不具有 MAC 地址。

将包存入缓冲区后，接下来需要查询一下这个包的接收方 MAC 地址是否已经在 MAC 地址表中有记录了。

交换机的 MAC 地址表主要包含两个信息：

  1. 一个是设备的 MAC 地址，
  2. 另一个是该设备连接在交换机的哪个端口上。

举个例子，如果收到的包的接收方 MAC 地址为 00-02-B3-1C-9C-F9，则与图中表中的第 3 行匹配，根据端口列的信息，可知这个地址位于 3 号端口上，然后就可以通过交换电路将包发送到相应的端口了。

所以，交换机根据 MAC 地址表查找 MAC 地址，然后将信号发送到相应的端口。

如果MAC表中没有对应IP地址的MAC地址，那么交换机就会广播所有端口，寻找MAC地址，如果找到了就会加入到MAC地址表中，如果没找到就会到让发送端主机到默认网关寻找MAC地址，接下来就是路由层不断的进行寻找，知道找到目标交换机，然后使用ARP协议找到MAC地址。

路由器

网络包经过交换机之后，现在到达了路由器，并在此被转发到下一个路由器或目标设备。

这一步转发的工作原理和交换机类似，也是通过查表判断包转发的目标。

不过在具体的操作过程上，路由器和交换机是有区别的。

因为路由器是基于 IP 设计的，俗称三层网络设备，路由器的各个端口都具有 MAC 地址和 IP 地址；

而交换机是基于以太网设计的，俗称二层网络设备，交换机的端口不具有 IP 地址。

路由器的端口具有 MAC 地址，因此它就能够成为以太网的发送方和接收方；同时还具有 IP 地址，从这个意义上来说，它和计算机的网卡是一样的。

当转发包时，首先路由器端口会接收发给自己的以太网包，然后路由表查询转发目标，再由相应的端口作为发送方将以太网包发送出去。
