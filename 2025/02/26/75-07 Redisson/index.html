73 Redisson

前言

如果你之前是在用 Redis 的话，那使用 Redisson 的话将会事半功倍，Redisson 提供了使用 Redis的最简单和最便捷的方法。

Redisson的宗旨是促进使用者对 Redis 的关注分离（Separation of Concern），从而让使用者能够将精力更集中地放在处理业务逻辑上。

Redisson 是一个在 Redis 的基础上实现的 Java 驻内存数据网格（In-Memory Data Grid）。

  • Netty 框架：Redisson采用了基于NIO的Netty框架，不仅能作为Redis底层驱动客户端，具备提供对Redis各种组态形式的连接功能，对Redis命令能以同步发送、异步形式发送、异步流形式发送或管道形式发送的功能，LUA脚本执行处理，以及处理返回结果的功能
  • 基础数据结构：将原生的Redis Hash，List，Set，String，Geo，HyperLogLog等数据结构封装为Java里大家最熟悉的映射（Map），列表（List），集（Set），通用对象桶（Object Bucket），地理空间对象桶（Geospatial Bucket），基数估计算法（HyperLogLog）等结构，
  • 分布式数据结构：这基础上还提供了分布式的多值映射（Multimap），本地缓存映射（LocalCachedMap），有序集（SortedSet），计分排序集（ScoredSortedSet），字典排序集（LexSortedSet），列队（Queue），阻塞队列（Blocking Queue），有界阻塞列队（Bounded Blocking Queue），双端队列（Deque），阻塞双端列队（Blocking Deque），阻塞公平列队（Blocking Fair Queue），延迟列队（Delayed Queue），布隆过滤器（Bloom Filter），原子整长形（AtomicLong），原子双精度浮点数（AtomicDouble），BitSet等Redis原本没有的分布式数据结构。
  • 分布式锁：Redisson还实现了Redis文档中提到像分布式锁Lock这样的更高阶应用场景。事实上Redisson并没有不止步于此，在分布式锁的基础上还提供了联锁（MultiLock），读写锁（ReadWriteLock），公平锁（Fair Lock），红锁（RedLock），信号量（Semaphore），可过期性信号量（PermitExpirableSemaphore）和闭锁（CountDownLatch）这些实际当中对多线程高并发应用至关重要的基本部件。正是通过实现基于Redis的高阶应用方案，使Redisson成为构建分布式系统的重要工具。

基础

什么是Redisson，它与Redis有什么关系？

有几个关键词：基于Redis开发的客户端，方便业务人员在分布式系统下开发，提供了分布式数据结构和工具，数据同步，数据共享，支持Redis的高可用。

答：Redisson是基于Redis的客户端，它对Redis进行了封装，方便业务人员在分布式环境下对Redis进行操作，为了实现分布式环境下的操作，它还提供了分布式锁、分布式集合等分布式数据结构和工具，这些都可以让在分布式环境下的服务达到数据共享和数据同步的目的。同时也支持Redis集群的高可用（哨兵模式、集群模式），方便业务人员快速的构建Redis集群，并自动的进行负载均衡和节点故障处理。

Redisson提供了哪些功能？

Redisson就是为了在分布式环境下为服务读取Redis中的数据而设置的，因为在多个服务下可能会出现数据竞争的问题，所以Redisson提供了分布式锁的概念，让同一时刻同一种方法只能有一个线程在运行，所以提供的第一个功能是分布式锁。

为了方便开发人员对分布式环境下对Redis数据读取，提供了大量的分布式数据结构，比如：RMap，RSet等等。

如果你具有多个Redis服务器，Redisson还可以帮你自动的管理这些分布式Redis，它自带了负载均衡和节点故障处理，可以方便业务人员快速的构建Redis集群。

Redisson是如何支持高可用的？

高可用是针对多个Redis情况下，如何保证Redis某个节点挂掉之后，整个服务依旧正常运行。这里就涉及到一些很常见的问题，即在集群环境下，如何保证整个集群的高可用。而Redisson是采用的什么方法呢？

在多个节点的情况下，我们无非需要知道当前节点是否可用以及当前节点故障之后应该将服务转发给哪个节点，如何保证多个节点之间的数据一致性，上面这三个问题就是我们重点需要关注的问题，那我们带着这三个问题来看看Redisson是如何解决的。

哨兵模式

哨兵简单的来说就是一个服务，这个服务时刻的监听Redis服务是否正常，一旦该Redis服务不正常，则需要立即更改Redis服务。

哨兵模式采用了多个Redis节点，它会选择其中一个作为Redis的主节点，其它Redis节点会作为子节点，也就是备胎。一旦哨兵发现该主节点Master发生故障，哨兵就会重新选择一个新的Master，然后通过发布/订阅模式告知其它子节点修改配置（新的Master节点变更）。

但是哨兵如果只有一个，那也是存在整个服务宕机的危险，所以哨兵也得多多的，越多越安全，鸡蛋不能放在一个笼子里嘛。

在这里要深挖的点太多了，比如：哨兵是如何实现的？如果主节点故障，哨兵是如何选取新的主节点？哨兵集群是如何设计的？如何保证Redis集群的数据一致性？哨兵通过发布-订阅模型通知其它子节点，它是如何实现的？

在很多时候要知其然更要知其所以然，下面将会详细的介绍。

哨兵是如何实现的

哨兵就是一个进程，哨兵集群就是一组进程，这组进程就干几件事：时不时的检查一下Redis节点是否还活着、选取新的Redis主节点，通知其它哨兵主节点故障大家一起来选新的主节点。

哨兵每隔一定时间会主动执行心跳检测，一旦主节点响应超时或断开，哨兵将其标记为“主观下线”。如果多个哨兵共同检测到主节点故障（达到quorum），该主节点将被标记为“客观下线”，这会触发故障转移。

当sentinel 哨兵节点将 master 标记为主观下线后，会向其余所有的 sentinel 发送sentinel is-master-down-by-addr消息，询问其他sentinel是否同意该master下线。

每个sentinel收到命令之后，会根据发送过来的 ip和port 检查自己判断的结果，回复自己是否认为该master节点已经下线了。

sentinel收到回复之后，如果同意master节点进入主观下线的sentinel数量大于等于quorum，则master会被标记为客观下线，即认为该节点已经不可用。

这就是哨兵啦。

如果主节点故障，哨兵是如何选取新的节点呢？

现在哨兵检测到主节点存在故障，需要马上转移主节点，接下来该如何操作呢？

我们哨兵现在很多呀，谁来主持这个选取新的主节点一事呢？这可是在程序世界，总要有一个程序去做这件事。在选取新的主节点之前，需要先把哨兵的临时话事人给选出来。怎么选？谁干？

这里就又又又又来了一个选举算法，称为Raft算法，它具体是怎么做的呢？

简单一句话，诶，谁最早发现这个主节点故障，谁就干这个脏活，话糙理不糙。

  1. 判断客观下线的sentinel节点向其他 sentinel 节点发送 SENTINEL is-master-down-by-addr ip port current_epoch runid

注意：这时的runid是自己的run id，每个sentinel节点都有一个自己运行时id

  1. 目标sentinel回复是否同意master下线并选举领头sentinel，选择领头sentinel的过程符合先到先得的原则。举例：sentinel1判断了客观下线，向sentinel2发送了第一步中的命令，sentinel2回复了sentinel1，说选你为领头，这时候sentinel3也向sentinel2发送第一步的命令，sentinel2会直接拒绝回复；
  2. 当sentinel发现选自己的节点个数超过 majority 的个数的时候，自己就是领头节点；
  3. 如果没有一个sentinel达到了majority的数量，等一段时间，重新选举；

选完干脏活的哨兵后，就要开始换主节点了，如何选取新的主节点呢？

Redis集群并不需要保证数据强一致性，所以只需要选取跟主节点最相似的子节点作为主节点即可，也就数据复制偏移量最少的子节点，同时优先级越高的越可能被设为主节点。

在选择的时候也会把那些可能出现故障的子节点排除候选人之列。

选出主节点之后，就需要进行权利交接咯。修改配置了。告知其它所有哨兵和子节点，这位就是我们新的主节点啦，要读就读它，监控它

如何保证Redis集群的数据一致性？

Redis 哨兵模式下的数据一致性主要通过以下方法实现：

  • 主从复制：从节点从主节点实时复制数据。在主节点故障并切换到从节点为新的主节点时，所有写操作将从新主节点处理。
  • 故障转移过程中的数据保护：在选举新主节点之前，哨兵会确保旧主节点的状态是可靠的，并且在选举过程中确保不会有写操作对旧主节点进行修改。
哨兵通过发布-订阅模型通知其它子节点

Redis 哨兵使用 发布-订阅模型（Pub/Sub）进行通知，主要通过以下机制：

  • 监控状态变化：每个哨兵会监控主从节点的状态，并在状态发生变化时通过发布-订阅模型将变化信息发送给其他哨兵和客户端。
  • SENTINEL MONITOR 命令：哨兵会订阅自己监控的节点状态。当主节点状态变化时，哨兵会发布相关事件，例如节点下线、提升新主节点等。

负载均衡机制

自动路由：

  • Redisson 会根据 Redis 哨兵提供的信息自动获取集群中各个节点的状态（包括主节点和从节点）。当发起请求时，Redisson 会将请求路由到合适的主节点（写操作）或从节点（读操作）。
  • 如果你配置了多主从结构，Redisson 可以平衡请求到不同的从节点，从而实现读操作的负载均衡。

轮询与随机策略：

  • Redisson 内置了多种负载均衡策略，可以在多个 Redis 节点之间分配请求。例如，可以使用轮询或随机策略均匀分配请求，从而有效利用各个节点的资源。

Redisson 提供了哪几种分布式锁？这些锁的使用场景是什么？其是如何实现的？

Redisson七种锁

参考资料：【实践篇】4.3 Redis之Redisson 7种分布式锁使用指南

简单锁
  • 简单锁是最基本的分布式锁，通过调用Redisson的getLock()方法来获取锁，并使用lock()和unlock()方法来加锁和解锁。
  • 简单锁在性能方面表现良好，可以提供良好的并发控制。
  • 普通的分布式锁：用于保护共享资源，防止并发操作引发的数据不一致问题。
  • 适用于大多数需要控制并发访问的场景，如数据库记录的更新、文件操作等。
公平锁
  • **公平锁是按照请求锁的顺序来获取锁的，先到先得。**通过调用Redisson的getFairLock()方法来获取公平锁。
  • 性能和公平性控制：公平锁在性能方面相对较低，因为需要维护一个请求锁的队列来保证公平性。

公平指的就是对于各个请求锁的线程来说是公平的，排队嘛，那就是公平的嘛，先到先得，客观公平。与之带来的问题就是性能较低了。

可重入锁
  • 可重入锁是可以被同一个线程多次获取的锁。通过调用Redisson的getReentrantLock()方法来获取可重入锁。
  • 性能和重入控制：可重入锁在性能方面表现良好，并且可以通过计数器来控制锁的重入次数。
  • 需要可重入的分布式锁：在同一线程中可以多次获取锁，适用于递归调用或多层方法调用的情况。

实现方式：

  • 通过在锁的值中存储线程标识符（如线程 ID），确保同一线程可以多次获取锁而不引发死锁。
  • 锁的计数器用于跟踪获取次数，只有在释放次数等于获取次数时，锁才会真正释放。
红锁
  • 红锁是在多个Redis实例上获取锁的一种机制，要求大多数Redis实例成功获取锁才算成功。通过调用Redisson的getRedLock()方法来获取红锁。
  • 性能和分布式锁控制：红锁在性能方面相对较低，因为需要在多个Redis实例之间进行同步。
读写锁
  • 读写锁是同时支持读操作和写操作的锁。通过调用Redisson的getReadWriteLock()方法来获取读写锁。
  • 读写锁在读操作方面允许并发访问，但在写操作方面只允许一个线程访问。
  • 高并发读场景：当读操作远远大于写操作时，可以使用读写锁来提高性能。
  • 适用于缓存、列表和共享数据的场景，例如读取配置文件的情况。
信号量
  • 信号量是一种用于控制并发访问数量的锁。通过调用Redisson的getSemaphore()方法来获取信号量。
  • 信号量可以控制同时访问的线程数量，从而控制并发访问。
  • 控制并发访问数量：适用于限制访问某些资源的线程数量，比如限流。
  • 适用于连接池、线程池等需要限制并发数的场景。
闭锁
  • 闭锁是一种用于等待其他线程完成任务的同步工具。通过调用Redisson的getCountDownLatch()方法来获取闭锁。
  • 闭锁可以等待其他线程完成任务后再继续执行。
  • 等待多个线程完成任务：适用于需要协调多个线程完成某项任务的场景。
  • 例如，主线程等待多个子线程完成后再继续执行。
