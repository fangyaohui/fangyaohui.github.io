概念介绍

Java Lambda表达式是Java 8中最重要的新特性之一。

它们是一种可传递的匿名函数，可以作为参数传递给方法或存储在变量中，因此可以在需要的时候调用它们。

Lambda表达式的主要目的是简化Java代码，使其更易于阅读和编写。

Lambda表达式的语法非常简洁和清晰。它们由参数列表、箭头符号和方法体组成。参数列表指定传递给Lambda表达式的参数，箭头符号 “->” 分隔参数列表和方法体，方法体则包含Lambda表达式要执行的代码。

Lambda优点

Lambda表达式的主要优点包括：

  1. 简化代码：Lambda表达式可以将冗长复杂的代码简化为几行简洁的代码。
  2. 可读性：Lambda表达式可以使代码更易于阅读和理解，因为它们更接近自然语言。
  3. 可传递性：Lambda表达式可以作为参数传递给方法或存储在变量中，使代码更具可重用性和灵活性。
  4. 并行处理：Lambda表达式可以与Stream API一起使用，使Java程序更容易地进行并行处理。

Lambda表达式是Java 8中最重要的新特性之一，它们为我们提供了一种更简单、更灵活、更易于使用的编程方式。

概述

Synchronized 【/ˈsɪŋkrənaɪzd/】是 Java 语言的一个关键字，它允许多个线程同时访问共享的资源，以避免多线程编程中的竞争条件和死锁问题。synchronized可以用来给对象或者方法进行加锁，当对某个对象或者代码块加锁时，同时就只能有一个线程去执行。这种就是互斥关系，被加锁的区域称为临界区，而里面的资源就是临界资源。当一个线程进入临界区的时候，另一个线程就必须等待。

1、通过对象进行锁

在代码里，可以通过创建一个对象，这样要想拿到临界资源，就必须先获得到这个对象的锁。

2、通过this

使用this代表锁住的是当前对象，这种方法等同直接把synchronized关键字加在方法前。

3、锁定静态方法

锁定静态方法需要通过类.class，或者直接在静态方法上加上关键字。但是，类.class不能使用this来代替。注：在同一个类加载器中，class是单例的，这也就能保证synchronized能够只让一个线程访问临界资源。

总结

  • A. 无论synchronized关键字加在方法上还是对象上，如果它作用的对象是非静态的，则它取得的锁是对象；如果synchronized作用的对象是一个静态方法或一个类，则它取得的锁是对类，该类所有的对象同一把锁。
  • B. 每个对象只有一个锁（lock）与之相关联，谁拿到这个锁谁就可以运行它所控制的那段代码。
  • C. 实现同步是要很大的系统开销作为代价的，甚至可能造成死锁，所以尽量避免无谓的同步控制。

Java基础

JVM vs JDK vs JRE

JVM (Java Virtual Machine)、JDK (Java Development Kit)、JRE (Java Runtime Environment) 是 Java 开发与运行环境中的三个重要概念。理解它们有助于全面掌握 Java 生态系统。下面将详细解释每个概念，并指出它们之间的区别。

1. JVM (Java Virtual Machine) - Java 虚拟机

JVM 是什么？

  • JVM 是一个虚拟机，负责将编译后的 Java 字节码（.class 文件）转换为机器码，并在特定平台上运行。这使得 Java 具有“平台无关性”的特性，即“写一次，运行到处”（Write Once, Run Anywhere）。

JVM 的主要功能：

  • 加载字节码：JVM 负责从 .class 文件中加载字节码。
  • 字节码解释与执行：将字节码解释为机器码，或通过即时编译 (JIT) 将其直接编译为机器码执行。
  • 内存管理：JVM 管理 Java 应用的内存（例如堆、栈、方法区等）。
  • 垃圾回收 (Garbage Collection)：自动清理不再使用的对象，避免内存泄漏。
  • 线程管理：JVM 也负责管理 Java 中的多线程操作。

JVM 不包括什么？

  • 它不包括编译工具和开发工具，单纯是执行 Java 字节码的环境。

2. JDK (Java Development Kit) - Java 开发工具包

JDK 是什么？

  • JDK 是 Java 开发者的工具包，包含了开发 Java 应用所需的一切。它包括 JRE、编译器 (javac)、调试工具、文档生成器等。

JDK 的主要组成部分：

  1. JRE (Java Runtime Environment)：用于运行 Java 应用，包含 JVM 和 Java 标准库。
  2. 编译器 (javac)：将 Java 源代码编译成字节码。
  3. 调试工具：如 jdb，用于调试 Java 程序。
  4. 文档工具 (javadoc)：生成 Java API 文档。
  5. 打包工具 (jar)：用于将 Java 应用程序打包成 .jar 文件。

JDK 的用途：

  • JDK 提供了所有开发 Java 应用程序所需的工具。开发者可以使用 JDK 编写、编译、调试和运行 Java 代码。

3. JRE (Java Runtime Environment) - Java 运行时环境

JRE 是什么？

  • JRE 是运行 Java 应用的环境。它包含 JVM 以及 Java 的类库和其他支持文件。JRE 是用户运行 Java 应用时所需的环境，而不是开发者写 Java 程序所需的工具包。

JRE 的主要组成部分：

  1. JVM：负责执行 Java 字节码。
  2. Java 标准库（类库）：如 java.lang、java.util、java.io 等，提供了 Java 程序常用的 API 和功能。
  3. 其他运行时支持文件：包括支持运行 Java 应用程序的资源文件等。

JRE 的用途：

  • JRE 是用户运行已经编译好的 Java 程序所需的最低环境，它不包括开发工具。对于非开发者，只要有 JRE，就可以运行 Java 应用。

它们之间的区别

总结

  • JVM 是 Java 的核心部分，它负责将 Java 字节码转换为机器码，并在特定平台上运行。
  • JRE 包含 JVM 以及 Java 标准库，用户只需要安装 JRE 就可以运行 Java 程序。
  • JDK 是开发者的工具包，它包含 JRE 和用于开发 Java 应用的工具，如编译器和调试工具。

总之，JVM 是运行时的核心，JRE 是 JVM 的运行环境，而 JDK 则是包含 JRE 和开发工具的完整开发套件。

为什么需要重写 hashCode()？

  1. 一致性原则：hashCode() 和 equals() 需要保持一致。如果两个对象通过 equals() 方法被认为相等，那么它们的 hashCode() 必须相同。如果不重写 hashCode()，即使两个对象在逻辑上是相等的，集合类如 HashSet 或 HashMap 仍会认为它们是不同的对象，从而引发问题。

  2. 效率问题：HashSet 和 HashMap 等基于哈希表的集合类使用 hashCode() 来快速确定对象的存储位置。如果没有重写 hashCode()，这些集合的性能会变得低下，因为它们需要依赖 equals() 进行逐一比较，而 hashCode() 可以大幅提高查找和存储效率。

总结

  • 重写 equals() 时必须重写 hashCode()，否则会导致集合类中的不一致性问题。
  • hashCode() 用于高效定位对象，而 equals() 用于精确比较对象的内容。

String、StringBuffer、StringBuilder的区别

线程安全

String 中的对象是不可变的，也就可以理解为常量，线程安全。 AbstractStringBuilder 是 StringBuilder 与 StringBuffer 的公共⽗类，定义了⼀些字符串的基本操作，如 expandCapacity 、 append 、 insert 、 indexOf 等公共⽅法。 StringBuffer 对⽅法加了同步锁或者 对调⽤的⽅法加了同步锁，所以是线程安全的。 StringBuilder 并没有对⽅法进⾏加同步锁，所以是 ⾮线程安全的。

名称 线程安全性
String 线程安全
StringBuffer 线程安全
StringBuilder 非线程安全

性能

每次对 String 类型进⾏改变的时候，都会⽣成⼀个新的 String 对象，然后将指针指向新的 String 对象。 StringBuffer 每次都会对 StringBuffer 对象本身进⾏操作，⽽不是⽣成新的对象并改 变对象引⽤。相同情况下使⽤ StringBuilder 相⽐使⽤ StringBuffer 仅能获得 10%~15% 左右的性 能提升，但却要冒多线程不安全的⻛险。

对于三者使⽤的总结：

  1. 操作少量的数据: 适⽤ String
  2. 单线程操作字符串缓冲区下操作⼤量数据: 适⽤ StringBuilder
  3. 多线程操作字符串缓冲区下操作⼤量数据: 适⽤ StringBuffer

String为什么是不可修改的

这是因为保存String的数组被final修饰且私有，对应类也没有提供/暴露操作的方法;String类被final修饰导致其不能被继承，进而避免了子类破坏String不可变。

从字节码⻆度分析 try catch finally 这个语法糖背后的实现原理。

从字节码层面来看，try-catch-finally 通过以下方式实现：

  1. 控制流跳转：JVM 使用跳转指令（如 goto）来确保无论在 try 块中发生什么（异常或正常完成），都可以在合适的时机跳转到 catch 和 finally 块。
  2. 异常表：JVM 并不是在每个字节码指令后面都插入异常处理代码，而是通过异常表来描述代码段之间的关系。异常表指明了在某个范围内发生特定类型的异常时，应该跳转到哪个字节码位置处理。
  3. finally 的执行保证：无论是否发生异常，finally 块总会被执行。字节码通过跳转（goto 指令）和异常表，确保了在 try 块和 catch 块之后，都会跳转到 finally 块中。

4. 重要结论

  • 异常表 是 try-catch-finally 的核心，它指明了异常发生时该跳转到哪里进行处理。
  • 控制流跳转 指令（如 goto）确保了异常发生后会进入 catch，并最终进入 finally，从而保证 finally 总是被执行。
  • 无论异常是否发生，finally 都会执行，这是由字节码中 goto 和异常表共同保证的。

为什么String在Java中被设计为不可变？

从安全性的角度来看，这是为了确保系统的安全性，比如String类会频繁的出现在数据库连接、文件地址URL等方面，如果可变的话，攻击可以通过恶意改变其值来达到破坏系统安全性的目的；

从线程安全性的角度来看的话，因为String的不可变，在多线程环境下，就无需考虑同步问题；

从哈希码缓存的角度来看，因为哈希是通过对String内容计算得出来的，无论是hashset还是hashmap，如果String可变，那么其修改String后其对应的哈希码会发生改变，进而影响后续集合的行为；

从内存的角度来看，这样可以节约内存的使用，因为String的不可变，所以String具有相同内容的就可以直接引用，而不用担心其String发生更改；

回答这个问题可以从：系统安全性、节约内存、线程安全性以及哈希缓存角度继续分析回答。

1. 泛型的背景与目的

在 Java 中，泛型是为了增强类型安全和提高代码重用性而引入的。泛型允许我们在编译时指定类型，而不是在运行时使用原始类型，这有几个明显的好处：

  • 类型安全：编译时检查，避免了运行时出现类型转换异常。
  • 代码复用：通过一个类或方法处理不同类型的数据，而不需要重复编写多个版本。

2. 泛型擦除是什么？

泛型擦除是 Java 泛型机制中的一个特性。简单来说，泛型擦除就是编译器在 编译时 会将所有泛型类型擦除（移除），并替换为它们的原始类型（raw types）。这意味着在 编译后，所有泛型类型都不再存在，JVM 运行时是无法知道泛型的具体类型的。

3. 为什么要做泛型擦除？

  • 向后兼容：为了兼容旧版本的 Java。Java 的泛型是在 Java 5 中引入的，但为了保持与 Java 1.4 及以前版本的兼容性，泛型信息在编译时会被移除，这样旧版 JVM 仍然可以运行新代码。
  • 性能考虑：在运行时，JVM 不需要维护泛型的类型信息，这减轻了运行时的开销。

4. 泛型擦除的实现

在 Java 中，泛型的实现是通过 类型擦除 来完成的。具体来说：

  • 泛型类型参数会被替换为 原始类型（raw type），通常是 Object，除非指定了泛型的上界（例如 List<T extends Number>）。
  • 你不能直接获取泛型参数的类型，因为编译后的字节码并没有包含这些信息。

5. 擦除带来的影响

由于擦除的存在，Java 运行时无法直接访问泛型的类型参数。这会导致一些常见的限制：

  • 无法进行 instanceof 检查：无法检查泛型类型的实例。

  • 泛型类型参数的信息丢失：在运行时无法知道泛型的具体类型。

  • 无法创建泛型数组：由于泛型信息在运行时丢失，不能创建如 new T[10] 的泛型数组。

6. 如何解决泛型擦除带来的限制

尽管泛型擦除存在上述限制，Java 提供了一些方法来克服这些问题：

  • 通配符（Wildcard）：可以使用 ? 来表示不确定的类型，例如 List<?> 表示任何类型的列表。
  • 反射：在某些情况下，可以利用反射来获取泛型信息，但是要注意，泛型信息只有在声明时可用，运行时不可得。

7. 泛型擦除与上界

如果泛型有上界，擦除会根据上界来决定原始类型。例如，如果你声明一个泛型类为 T extends Number，则擦除后的类型是 Number，而不是 Object。

在这种情况下，T 被擦除为 Number。

8. 总结

  • 泛型擦除 是 Java 编译器的特性，在编译阶段将泛型信息擦除，保留原始类型。这是为了保证向后兼容和优化性能。
  • 擦除后的结果：在编译后，泛型被替换为原始类型，且运行时不再保留泛型的类型信息。
  • 影响：无法进行类型检查、无法获取泛型类型参数等，但可以通过通配符和反射等手段克服这些限制。
