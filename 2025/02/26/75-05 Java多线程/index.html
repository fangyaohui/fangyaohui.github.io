75-05 Java多线程

多线程基础

线程 ＆ 进程

进程：是系统资源调度和分配的基本单位，操作系统分配的内存等一些系统资源都是直接分配给进程的。

线程：是CPU调度和分配的基本单位，线程一定属于一个进程，线程从进程中获取所需要的内存等其它一些系统资源。

多线程带来的问题

线程安全

在单线程下正常执行的代码可能在高并发的情况下并不安全，这是因为有些代码不是原子性的，就是在执行代码的时候可能会执行到一半再去执行其它的代码。

还有就是可见性问题，就是我线程A对数据进行了更新，但是其只是在线程内进行了更新，并没有即时更新到主内存中，基于JVM来说就是指，线程只是把数据保存到了Java虚拟机栈中，还未来得及把数据更新到堆上。这样就可能导致线程B来读取这个数据并不是最新的，这就是可见性问题。

在高并发情况下也可能出现多线程占有然后抢占资源的情况导致死锁，还可能存在多线程互相礼让导致活锁问题以及线程一直得不到运行的饥饿问题。

Java线程的6种状态

我们先来看看操作系统中的线程状态转换。在操作系统中，线程被视为轻ᰁ级的进程，所以线程状态其实和进程状态是⼀致的。

来看看在JVM中的线程的状态是什么样的。

NEW

处于 NEW 状态的线程此时尚未启动。这⾥的尚未启动指的是还没调⽤ Thread 实例的 start() ⽅法。

关于start有两个比较经典的问题

  1. 反复调用同一个现成的start方法是否可行？
  2. 假如一个线程执行完毕（此时处于 TERMINATED 状态），再次调⽤这个线程的 start ⽅ 法是否可⾏？

答案是都不行。

看看start源码就能知道答案。

可以看到start的源码中有一个ThreadStatus记录着这个线程的状态，如果这个线程被启动过就都不会为0，一旦该状态不为0而又启动了start就会抛出异常。

RUNNABLE

表示当前线程正在运⾏中。处于 RUNNABLE 状态的线程在 Java 虚拟机中运⾏，也有可能在 等待 CPU 分配资源。

也就是说，Java 线程的RUNNABLE状态其实包括了操作系统线程的ready和running两个状态。

BLOCKED

阻塞状态。处于 BLOCKED 状态的线程正等待锁（锁会在后⾯细讲）的释放以进⼊同步区。 我们⽤ BLOCKED 状态举个⽣活中的例⼦： 假如今天你下班后准备去⻝堂吃饭。你来到⻝堂仅有的⼀个窗⼝，发现前⾯已经有个⼈在窗⼝ 前了，此时你必须得等前⾯的⼈从窗⼝离开才⾏。 假设你是线程 t2，你前⾯的那个⼈是线程 t1。此时 t1 占有了锁（⻝堂唯⼀的窗⼝），t2 正在 等待锁的释放，所以此时 t2 就处于 BLOCKED 状态。

WAITING

等待状态。处于等待状态的线程变成 RUNNABLE 状态需要其他线程唤醒。

TIMED_WAITING

超时等待状态。线程等待⼀个具体的时间，时间到后会被⾃动唤醒。

TERMINATED

终⽌状态。此时线程已执⾏完毕。

线程组

ThreadGroup 是⼀个标准的向下引⽤的树状结构，这样设计可以防⽌”上级”线程被”下级”线程 引⽤⽽⽆法有效地被 GC 回收。

ThreadGroup 和 Thread 的关系就如同他们的字⾯意思⼀样简单粗暴，每个 Thread 必然存在 于⼀个 ThreadGroup 中，Thread 不能独⽴于 ThreadGroup 存在。执⾏ main() ⽅法的线程 名字是 main，如果在 new Thread 时没有显式指定，那么默认将⽗线程（当前执⾏ new Thread 的线程）线程组设置为⾃⼰的线程组。

总结⼀下，线程组是⼀个树状的结构，每个线程组下⾯可以有多个线程或者线程组。线程组可 以起到统⼀控制线程的优先级和检查线程权限的作⽤。

线程优先级可以指定，范围是 1~10。但并不是所有的操作系统都⽀持 10 级优先级的划分 （⽐如有些操作系统只⽀持 3 级划分：低、中、⾼），Java 只是给操作系统⼀个优先级的参 考值，线程最终在操作系统中的优先级还是由操作系统决定。 Java 默认的线程优先级为 5，线程的执⾏顺序由调度程序来决定，线程的优先级会在线程被 调⽤之前设定。 通常情况下，⾼优先级的线程将会⽐低优先级的线程有更⾼的概率得到执⾏。 Thread 类的 setPriority() ⽅法可以⽤来设定线程的优先级。

Java 中的优先级不是特别的可靠，Java 程序中对线程所设置的优先级只是给操作系统⼀个建 议，操作系统不⼀定会采纳。⽽真正的调⽤顺序，是由操作系统的线程调度算法来决定的。

还有⼀种特殊的线程，叫做守护线程（Daemon），守护线程默认的优先级⽐较低。

  • 如果某线程是守护线程，那如果所有的⾮守护线程都结束了，这个守护线程也会⾃动结 束。

  • 当所有的⾮守护线程结束时，守护线程会⾃动关闭，这就免去了还要继续关闭⼦线程的麻 烦。

  • 线程默认是⾮守护线程，可以通过 Thread 类的 setDaemon ⽅法来设置为守护线程。

所以，如果某个线程的优先级⼤于线程所在线程组的最⼤优先级，那么该线程的优先级将会失 效，取⽽代之的是线程组的最⼤优先级。

线程组的优先级限定了这个线程组的线程优先级最高为多少。

多线程带来的问题

Java内存模型

Java 内存模型（Java Memory Model，JMM）定义了 Java 程序中的变量、线程如何和主存 以及⼯作内存进⾏交互的规则。它主要涉及到多线程环境下的共享变ᰁ可⻅性、指令᯿排等问 题，是理解并发编程中的关键概念。

并发编程的线程之间存在两个问题：

  1. 两个线程之间是如何进行通信的？线程之间是以何种机制来交换信息的；
  2. 线程之间是如何进行同步的？线程之间是以何种机制来控制线程间发生的先对顺序；

有两种并发模型可以解决这两个问题：

  1. 消息传递并发模型
  2. 共享内存并发模型

Java 使⽤的是共享内存并发模型。

对于每⼀个线程来说，栈都是私有的，⽽堆是共有的。

从图中可以看出：

  1. 所有的共享变ᰁ都存在主存中。
  2. 每个线程都保存了⼀份该线程使⽤到的共享变ᰁ的副本。
  3. 如果线程 A 与线程 B 之间要通信的话，必须经历下⾯ 2 个步骤：
    1. 线程 A 将本地内存 A 中更新过的共享变ᰁ刷新到主存中去。
    2. 线程 B 到主存中去读取线程 A 之前已经更新过的共享变ᰁ。

注意，根据 JMM 的规定，线程对共享变量的所有操作都必须在⾃⼰的本地内存中进⾏，不能 直接从主存中读取。

内存可见性

因为Java内存模型规定，所有线程对于共享变量的所有操作必须在本地内存中进行，对于操作系统中的一二级缓存中进行操作，那么如果对于同一个数据，就会分别加载到各个的线程内存中。那么如何知道这个共享变量被其它线程更新了呢？

这里的部分内容在操作系统中保证CPU缓存一致性的时候会介绍部分内容。

JMM 说白了就是定义了一些规范来解决这些问题，开发者可以利用这些规范更方便地开发多线程程序。对于 Java 开发者说，你不需要了解底层原理，直接使用并发相关的一些关键字和类（比如 volatile、synchronized、各种 Lock）即可开发出并发安全的程序。

可见性简单来说，就是在多线程环境下操作相同数据时可以保证任何一个线程对这个数据修改其它线程都是可见的。

内存原子性

原子性简单来说就是在多线程环境下操作同一数据时保证只能有一个线程来进行操作，即保证互斥。

volatile 关键字

volatile 可以保证可⻅性，但不保证原⼦性：

  • 当写⼀个 volatile 变量时，JMM 会把该线程在本地内存中的变ᰁ强制刷新到主内存中 去；
  • 这个写操作会导致其他线程中的 volatile 变量缓存⽆效。

volatile会禁止指令重排。

synchronized关键字

在 Java 中，关键字 synchronized 可以保证在同⼀个时刻，只有⼀个线程可以执⾏某个⽅法 或者某个代码块(主要是对⽅法或者代码块中存在共享数据的操作)，同时我们还应该注意到 synchronized 的另外⼀个᯿要的作⽤，synchronized 可保证⼀个线程的变化(主要是共享数据 的变化)被其他线程所看到（保证可⻅性，完全可以替代 volatile 功能）。

synchronized保证了可见性和原子性。

synchronized 关键字最主要有以下 3 种应⽤⽅式：

  1. 同步⽅法，为当前对象（this）加锁，进⼊同步代码前要获得当前对象的锁；
  2. 同步静态⽅法，为当前类加锁（锁的是 Class 对象），进⼊同步代码前要获得当前类的 锁；
  3. 同步代码块，指定加锁对象，对给定对象加锁，进⼊同步代码库前要获得给定对象的锁。

synchronized属于可重⼊锁 从互斥锁的设计上来说，当⼀个线程试图操作⼀个由其他线程持有的对象锁的临界资源时，将 会处于阻塞状态，但当⼀个线程再次请求⾃⼰持有对象锁的临界资源时，这种情况属于᯿⼊ 锁，请求将会成功。

synchronized的四种锁状态

⾸先需要明确的⼀点是：Java 多线程的锁都是基于对象的，Java 中的每⼀个对象都可以作为 ⼀个锁。 还有⼀点需要注意的是，我们常听到的类锁其实也是对象锁，上⼀节我们也讲到了，应该有不 少⼩伙伴注意到了。 这⾥再多说⼏句吧。Class 对象是⼀种特殊的 Java 对象，代表了程序中的类和接⼝。Java 中 的每个类型（包括类、接⼝、数组以及基础类型）在 JVM 中都有⼀个唯⼀的 Class 对象与之 对应。这个 Class 对象被创建的时机是在 JVM 加载类时，由 JVM ⾃动完成。
