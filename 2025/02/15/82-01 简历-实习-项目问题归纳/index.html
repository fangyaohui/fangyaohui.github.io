82-01 简历-实习-项目问题归纳

实习-代码评审

描述

下面来稍微详细的描述一下整个代码评审的流程，以一个新仓库加入代码评审功能为例。例如项目有一个部门中的其中一个代码库希望添加这个功能，那么在流水线那边会配置好提交代码后需要触发调用我们这边本地的一个接口，只负责触发，不关心对方是否接受到。

我们这边收到了提交代码的请求，如果这个代码库是最新加入的代码库的话，则需要先对这个代码库进行构建知识图谱。我们需要获取到这个代码库中的相关类、方法和调用关系，比如利用AST解析工具或者JavaParaser工具。

JavaParser 是一个开源的 Java 源代码解析器和操作库，它能够解析 Java 源代码并构建对应的抽象语法树（AST）。JavaParser 提供了一组 API，使得开发者可以轻松地分析、修改和生成 Java 源代码。据此，你可以通过这些API来实现Java动态编译，原生的java库有javax.tools,但可惜不能支持对于AST的修改

还有比如相应的Git log、测试代码（可能需要手动提供，功能具体未实现）以及一些具体部门中的特定的业务需求。然后使用Neo4j来存储知识图谱。

你的问题非常专业，涉及 大模型如何利用知识图谱 以及 代码库知识图谱的维护与优化，我会结合实际场景进行详细解答。

问题 3：是否每次提交都需要更新知识图谱？如何优化？

问题分析

  • 如果每次提交都更新

    ，可能导致：

    • 许多变更 不影响调用关系，但仍然被分析，增加不必要的开销。

  1. 按需更新

问题 4：不同项目是否需要单独的知识图谱？如何管理多个知识图谱？

场景

  • 不同项目的代码库，是否应该分别构建知识图谱？
  • 同一部门的多个项目，能否共用一个知识图谱？

两种选择

方案 1：每个代码库一个独立知识图谱

  • 优点：
    • 隔离性强，不会混淆不同项目的调用关系。
  • 缺点：
    • 管理复杂，大模型查询时需要知道用哪个图谱。

    • 减少重复数据（若多个项目共用同一方法，可以共享节点）。

  • 缺点

    ：

    • 需要对大模型做 上下文增强，以便它知道查询哪个项目的知识图谱。

总结

  1. 让大模型使用知识图谱：
    • 使用 RAG 技术，大模型在回答前自动查询知识图谱，减少用户 prompt 复杂度。
  2. 代码库知识图谱变更后如何更新：
    • 监听代码提交，增量更新 AST，只修改受影响的调用关系。
  3. 优化更新效率：
    • 仅解析变更的文件，使用 增量 AST 解析，避免全量重建知识图谱。
  4. 多个代码库的知识图谱管理：
    • 可以 每个代码库单独维护，也可以 统一管理，按项目分隔，查询时自动筛选。

这套方案能在 保证高效更新 的同时 减少大模型的 prompt 依赖，并且适用于 大规模代码库的分析与维护。

代码评审实习项目涉及到的难点

项目背景：现在你在一家公司开发了一个代码评审相关的系统，你获取到用户的变更代码后，你需要获取到这个变更代码的上下文，然后与prompt一起包装发给大模型进行评审。

这里其中涉及到的难点有：

  1. 代码上下文提取；
  2. Prompt优化；
  3. 大模型输入长度控制；

代码审查哪些方面的内容

首先的第一点是你的prompt代码审查主要是评审哪些方面的问题：

  1. 首先就是代码中的一些常见问题，例如安全方面的问题，比如是否存在SQL注入的风险，xss攻击，未处理的异常等情况；
  2. 其次就是是否存在逻辑性错误，比如是否存在潜在的逻辑错误或者边界情况未考虑到；
  3. 风格一致性检查，是否符合项目的代码规范，比如一些命名方式等；
  4. 还有就是代码质量，代码是否符合最佳实践？是否存在冗余等情况；

Prompt优化

面试官询问：你是如何进行prompt优化的。关于Prompt优化紧扣下面几个技术点，分别为Prompt 提示工程、few shot示例引导以及CoT三大技术点进行Prompt优化。

Prompt 提示工程：比如设定角色，明确任务，详细的告知大模型做什么，从而减少幻觉。同时提供相关变更代码的必要的上下文信息，让大模型捕获更多的背景信息。

添加Few-shot示例：在Prompt中添加部分具有代表性的案例，让大模型学习模式，从而提高评审效率。

任务拆解（CoT）：如果直接让LLM进行评审，可能导致跳步，导致结果并不准确，CoT可以将任务分成多个小任务，从而引导大模型进行分析回答。

代码上下文获取

面试可能针对代码上下文获取来进行询问，比如在你这个项目中是如何获取到当前变更代码的上下文信息的呢？

实习-浦东智慧机场

前言

在刚刚的面试中，面试官主要考察了以下几个技术难点和知识点：

1. 大数据量处理

  • 面试官考察了你如何处理海量数据，尤其是针对浦东智慧机场项目中的大规模旅客数据。
  • 考察点包括：
    • 如何进行数据分层架构设计（OLTP、OLAP等）。
    • 如何进行分库分表及流式计算处理。
    • 使用Kafka等消息队列来保障数据的实时性和高吞吐量。

2. 数据库选型

  • 面试官考察了你在处理大数据时的数据库选型，并且要求你阐述具体的存储方案及原因。
  • 考察点包括：
    • 对于事务型数据使用了哪些数据库（MySQL、TiDB等）。
    • 对于分析型数据选择了哪些数据库（ClickHouse、Doris等）。
    • 缓存解决方案（Redis）以及如何减少数据库压力。

3. 第三方数据平台适配与幂等性保证

  • 面试官考察了你如何设计多次请求及幂等性保证，尤其是在高并发的情况下保证数据的正确性和一致性。
  • 考察点包括：
    • 如何使用**唯一标识（如request_id）**确保幂等性。
    • 设计如何保证数据不会重复插入或丢失。
    • 重试机制和如何应对请求失败（如网络问题、接口挂掉等）。

4. 网络异常及请求失败处理

  • 面试官考察了你在面对请求失败的情况下的应对策略，尤其是在网络异常的情况下如何保证系统的可靠性和数据完整性。
  • 考察点包括：
    • 指数退避重试机制，减少系统压力。
    • 熔断与限流策略，避免因单一接口故障导致整个系统崩溃。
    • 日志监控和降级方案设计，保证服务高可用性。

总结

这些问题考察了你在高并发系统设计、大数据量存储与处理、分布式架构设计以及数据一致性和系统可靠性等方面的知识，尤其是在处理复杂项目中的实际问题时，如何做出合理的技术决策和应对策略。
