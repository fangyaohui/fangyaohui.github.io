82-01 简历-实习-项目问题归纳

实习-代码评审

描述

下面来稍微详细的描述一下整个代码评审的流程，以一个新仓库加入代码评审功能为例。例如项目有一个部门中的其中一个代码库希望添加这个功能，那么在流水线那边会配置好提交代码后需要触发调用我们这边本地的一个接口，只负责触发，不关心对方是否接受到。

变更单服务接收到了一个评审任务请求之后，对这个任务进行简单处理后，比如获取当前代码库以及提交的patch。当前任务处理完成之后将任务提交至消息队列，然后上下文服务从消息队列中消费消息，获取当前变更代码段的上下文，然后将处理后的结果提交至消息队列，然后AI评审服务从消息队列中消费消息，获取消息之后做一定的处理之后提交给大模型进行评审。

各个服务之前通过消息中间件进行通信，那么我们使用的消息中间件是Redis。我们会使用多个消费者和多个生产者的模型来进行处理，首先第一个问题就是你如何保证Redis作为消息队列中的消息不回丢失呢？如何保证Redis中的消息不会被重消费呢？如何去确保Redis的可靠性呢？

存在下面的这些问题：

  1. 在这个服务里使用到了消息队列，那么请问你为什么选择使用Redis作为你们项目中的中间件呢？为什么不选择目前比较成熟的消息中间件，比如：Kafka和RabbitMQ等呢？

    选择Redis的核心原因：

    • 技术栈统一性：团队已有成熟的Redis基础设施（集群、监控、运维经验），复用现有中间件降低学习成本和运维复杂度。
    • 轻量级场景适配：代码评审任务属于低频、非强顺序性、消息体较小的场景（代码diff通常为KB级别），Redis内存操作的高吞吐量（10万级QPS）完全能满足需求。
    • 实时性要求：评审任务需要快速触发（秒级延迟），Redis的Pub/Sub或Streams模式在低延迟场景优于Kafka的批处理模型。
    • 简化架构：避免引入新的中间件依赖，减少系统复杂度。Kafka需要维护ZooKeeper集群，RabbitMQ需要维护Erlang环境，而Redis已是现有基础设施。

    规避Kafka/RabbitMQ的考量：

    • Kafka的强顺序性和分区特性在此场景中无必要，且其设计更适合日志类大数据量场景。
    • RabbitMQ的复杂路由（Exchange/Binding）和ACK机制在此处属于过度设计，且AMQP协议的解析开销较高。

  2. 那么你选择使用Redis作为你的中间件，它会存在哪些问题？你们又是如何去解决这个问题的呢？

    1、消息堆积：由于消费者宕机或者消费速度过慢导致消息被消费的很慢，导致内存溢出；

    2、持久化可靠性：Redis作为消息中间件，需要对消息进行持久化操作，使用RDB可能丢失部分数据，但是使用AOF其性能会有所损失；

    3、无原生的重试队列：无超时自动重传，需要自己手动管理；

    4、不保证消息不丢失：Redis pub/sub机制无ACK机制，Stream需要自行实现；

    解决方案：

    1、选择使用Redis中的Stream数据结构，而非使用pub/sub，同时对Redis内存设置阈值，超过指定的阈值触发警告，同时限制消息队列的长度，也就是Stream长度；

    2、针对Redis持久化问题，因为Redis如果选择使用RDB来进行持久化的话，因为是异步对当前创建快照的那一刻的Redis进行刷屏，所以会导致中间的这段时间的数据可能存在丢失问题，但是AOF性能较差，所以可以采用两者的混合式，对于创建快照的那一刻的数据选择使用RDB，而对于后面新增的数据选择使用AOF；

  1. 首先就是代码中的一些常见问题，例如安全方面的问题，比如是否存在SQL注入的风险，xss攻击，未处理的异常等情况；
  2. 其次就是是否存在逻辑性错误，比如是否存在潜在的逻辑错误或者边界情况未考虑到；
  3. 风格一致性检查，是否符合项目的代码规范，比如一些命名方式等；
  4. 还有就是代码质量，代码是否符合最佳实践？是否存在冗余等情况；

Prompt优化

面试官询问：你是如何进行prompt优化的。关于Prompt优化紧扣下面几个技术点，分别为Prompt 提示工程、few shot示例引导以及CoT三大技术点进行Prompt优化。

Prompt 提示工程：比如设定角色，明确任务，详细的告知大模型做什么，从而减少幻觉。同时提供相关变更代码的必要的上下文信息，让大模型捕获更多的背景信息。

添加Few-shot示例：在Prompt中添加部分具有代表性的案例，让大模型学习模式，从而提高评审效率。

任务拆解（CoT）：如果直接让LLM进行评审，可能导致跳步，导致结果并不准确，CoT可以将任务分成多个小任务，从而引导大模型进行分析回答。

面试可能针对代码上下文获取来进行询问，比如在你这个项目中是如何获取到当前变更代码的上下文信息的呢？

RAG

RAG是一个检索、增强和生成的AI框架，它可以为大模型提供更多的外部信息，通过RAG检索获取到更多的信息之后拼接到promte后交给大模型进行处理，那么大模型获取到的上下文信息会更多。

RAG的“检索、增强、生成”，谁增强了谁，谁生成了答案，主语很重要。是从知识库中检索到的问答对，增强了LLM的提示词（prompt），LLM拿着增强后的Prompt生成了问题答案。

如何保证请求接口的幂等性

好的，我现在需要回答如何保证接口的幂等性这个问题。首先，我得理解什么是幂等性。幂等性是指无论调用接口多少次，结果都是一致的。比如，用户多次点击提交订单，系统应该只生成一个订单，而不是多个。这在实际开发中非常重要，尤其是在网络不稳定的情况下，用户可能会重复发送请求。

接下来，我需要回忆常见的保证接口幂等性的方法。首先想到的是使用唯一标识符，比如订单ID或者请求ID。每次请求时生成一个唯一的ID，服务端检查这个ID是否已经处理过，如果处理过就直接返回之前的结果，避免重复处理。这种方法需要服务端存储这些ID，可能会用到数据库或者缓存，比如Redis。

然后是乐观锁。比如在更新数据的时候，带上一个版本号，只有版本号匹配的时候才执行更新，这样可以防止重复更新。比如在扣减库存的时候，先读取当前库存和版本号，然后更新时检查版本号是否一致，如果不一致说明有其他请求已经修改过，就拒绝当前请求。

还有一种方法是Token机制。客户端在发起请求前先获取一个Token，服务端生成并保存这个Token，之后客户端的请求必须携带这个Token，服务端处理完请求后删除Token。这样重复的请求会因为Token不存在而被拒绝。

还有数据库的唯一约束，比如在表中设置唯一索引，防止插入重复的数据。比如用户注册时，用户名必须唯一，设置唯一索引后，重复插入会抛出异常，从而保证幂等性。

另外，状态机也是一种方法。业务操作按照一定的状态流转进行，比如订单状态从“待支付”到“已支付”，如果收到重复的支付请求，系统会检查当前状态，如果是“已支付”就直接返回成功，不再处理。

现在需要考虑用户提到的场景，调用第三方接口且数据量大需要分批次请求，可能存在失败重试。这时候需要确保即使多次重试，数据库的结果一致。可能需要结合唯一批次ID和状态记录，每次处理前检查批次状态，如果已经处理过就直接跳过，否则处理并更新状态。

还要考虑分布式系统中的问题，比如网络延迟、服务宕机等。可能需要引入分布式锁或者使用消息队列来保证顺序和唯一性。

不过，用户在问题中提到了调用第三方接口，可能存在失败的情况，所以需要设计重试机制，并且在重试时保证幂等性。比如每个批次请求有一个唯一ID，第三方接口支持幂等性，或者本地记录每个批次的状态，避免重复提交。

还需要注意数据一致性问题，比如在部分批次成功、部分失败的情况下，如何回滚或补偿。可能需要事务机制或者最终一致性方案。

总结一下，保证接口幂等性的方法包括唯一ID、Token机制、乐观锁、数据库唯一约束、状态机等。具体实现需要根据业务场景选择合适的方法，并结合重试机制和状态管理来确保数据一致性。

保证接口的幂等性可以通过以下方法实现：

1. 唯一请求标识（Request ID）

• 实现方式：客户端在每次请求时生成一个全局唯一的请求ID（如UUID），服务端通过该ID判断请求是否重复。
• 步骤：

  1. 客户端生成唯一ID，并随请求发送至服务端。
  2. 服务端在Redis或数据库中记录该ID的状态（如已处理/未处理）。
  3. 处理请求前，服务端检查该ID是否已存在：
    ◦ 若存在，直接返回之前的处理结果。
    ◦ 若不存在，处理请求并标记ID为已处理。
    • 适用场景：支付、订单提交等高并发写操作。

2. Token 机制

• 实现方式：客户端在执行业务操作前，先申请一个一次性Token。
• 步骤：

  1. 客户端调用预接口（如/api/get_token）获取Token。
  2. 服务端生成Token并存储（如Redis，设置短时过期）。
  3. 客户端携带Token发起正式请求。
  4. 服务端验证Token是否存在：
    ◦ 存在：处理请求并删除Token。
    ◦ 不存在：拒绝请求（视为重复提交）。
    • 适用场景：表单提交、抢购等需防止重复提交的场景。

3. 数据库唯一约束

• 实现方式：利用数据库的唯一索引或联合唯一键，避免重复数据插入。
• 步骤：

  1. 设计业务表时，对关键字段（如订单号、用户ID+活动ID）添加唯一索引。
  2. 插入数据时，若触发唯一冲突，直接捕获异常并返回已有结果。

4. 乐观锁（版本号控制）

• 实现方式：为数据添加版本号字段，更新时校验版本是否一致。
• 步骤：

5. 状态机（业务状态流转）

• 实现方式：定义业务状态流转规则，拒绝非法状态变更。
• 步骤：

  1. 设计状态流转图（如订单状态：待支付 → 已支付 → 已完成）。
  2. 处理请求时，检查当前状态是否允许变更：
    ◦ 若允许，更新状态并执行业务逻辑。
    ◦ 若不允许，直接返回当前结果。

6. 分布式锁

• 实现方式：在分布式环境下，通过锁机制确保同一时间仅一个请求能执行业务逻辑。
• 步骤：

  1. 使用Redis或ZooKeeper实现分布式锁（如SET lock_key 1 NX EX 30）。
  2. 获取锁后执行业务逻辑，完成后释放锁。
  3. 若获取锁失败，直接返回提示或等待重试。
    • 优化：锁粒度细化到资源ID（如lock:order:123），避免全局锁竞争。
    • 适用场景：秒杀、分布式任务调度等高并发场景。

7. 消息队列去重

• 实现方式：通过消息队列的ACK机制和唯一ID实现消费端幂等。
• 步骤：

  1. 消息生产者为每条消息生成唯一ID。
  2. 消费者处理消息前，检查该ID是否已处理：
    ◦ 若已处理，直接ACK确认。
    ◦ 若未处理，执行业务逻辑并记录ID。
  3. 结合数据库唯一键或Redis实现去重。
    • 适用场景：异步任务处理、大数据量批量操作。

8. 第三方接口协作

• 实现方式：若接口依赖第三方服务，需确保第三方支持幂等性。
• 步骤：

  1. 调用第三方接口时传递唯一ID（如X-Idempotency-Key: request_123）。
  2. 第三方根据该ID保证相同请求仅处理一次。
  3. 若第三方不支持幂等，本地需记录已成功请求的ID。
    • 示例：支付接口（如支付宝、微信支付通常支持幂等性）。

总结

• 简单场景：优先使用数据库唯一约束或乐观锁。
• 高并发场景：结合Redis唯一请求ID或分布式锁。
• 依赖第三方：确保传递唯一ID并利用其幂等性支持。
• 最终一致性：通过重试机制+状态机+异步补偿保障。

通过以上方法，可有效避免重复请求导致的脏数据或业务异常，确保系统健壮性。
